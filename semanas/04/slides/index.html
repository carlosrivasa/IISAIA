"Lo que vimos después es que esa predicción se mete en un loop. Y acá empieza algo importante: el loop no lo corre el LLM. Lo corre un programa afuera — la rutina del provider, del SDK, del runtime — que toma la salida del LLM, la concatena a la entrada, le vuelve a pedir el próximo token, y corta cuando se alcanza un tope explícito como max_tokens. Esa caja punteada es lo nuevo que tienen que ver."

"La versión madura del loop introduce dos usos de tokens especiales que el LLM aprendió en una etapa puntual de su entrenamiento: el supervised fine-tuning, o SFT. Por eso ahora la caja dice 'LLM + SFT'. El primer token especial, en la salida: durante SFT el modelo aprende a emitir un token especial — end-of-text — que el programa afuera interpreta como 'terminé'. La señal viene de adentro del LLM, pero quien la lee y detiene la generación sigue siendo el programa. El segundo, en la entrada: el usuario manda texto plano, sin nada raro. Es el programa el que envuelve ese texto con tokens de rol — pone `<|user|>` antes del mensaje y `<|assistant|>` al final, como diciéndole 'acá te toca a vos'. El LLM aprendió en SFT a leer esos marcadores y saber qué parte del texto es entrada y qué parte tiene que generar."

"El siguiente paso del entrenamiento agrega una capa más: reinforcement learning, RL. Por eso la caja dice 'LLM + SFT + RL'. Lo que aprende el modelo con RL — entre otras cosas — es a razonar antes de responder. Genera una cadena de pensamiento envuelta en tokens especiales como `<thinking>...</thinking>` que es trabajo interno del modelo, y después la respuesta final para el usuario. El programa externo lidia con esto: lee el output completo, filtra los tokens de razonamiento, y solo le muestra al usuario la respuesta final. El usuario ve un mensaje limpio; el pensamiento queda detrás de bambalinas. Otra responsabilidad nueva del programa, y otra cosa que el LLM aprendió en una etapa específica de entrenamiento."

"Ahora dejá que los cuatro se vayan y mirá esto solo. El programa afuera hace una cosa más que es lo que arma la dinámica de conversación que conocen. Tres bloques: el system prompt — un texto que el programa inyecta una sola vez al principio de la sesión y que el LLM va a tener delante en cada turno. El historial — todo lo que el usuario y el modelo se dijeron antes, que el programa guarda. Y el mensaje nuevo. El programa concatena los tres bloques en un solo texto y eso es lo que el LLM recibe. El LLM genera la respuesta — siempre con su loop, su `<END>` y su razonamiento interno — y el programa toma esa respuesta y la suma al historial. Próximo turno, el historial es un poco más largo. El LLM por sí solo no recuerda nada entre turnos; el que recuerda es el programa. La semana 4 entera trata sobre cómo evoluciona este programa: empezó contando tokens, aprendió a mantener una conversación, va a terminar siendo un agente."

Termina con <END>, fin. No hay loop, no hay decisión nueva del modelo después de ver los resultados. Y muchas tareas reales necesitan exactamente eso: decidir qué hacer en base a lo observado."

ReAct: cuando una vuelta no alcanza

El programa externo entra en un loop: el LLM piensa, actúa usando tools, observa los resultados, y decide la próxima acción. Vuelta a vuelta, hasta una condición de corte.

antes — una vuelta

El LLM puede llamar una o varias tools dentro de una sola respuesta y termina. El modelo no decide qué hacer con los resultados.

ahora — ReAct

Múltiples vueltas. Después de cada observación, el LLM piensa de nuevo y elige el próximo paso a partir de lo que vio.

Tres estados, dos reveals: arranca con título y lede, después aparecen animación → cards de contraste. Bridge con el slide anterior en el primer em.

Introducción de ReAct como la siguiente arquitectura del programa externo: en lugar de una respuesta con tools dentro, el LLM puede iterar — pensar, actuar (tools), observar el resultado, pensar de nuevo, actuar otra vez, hasta resolver la tarea. La diferencia con single-turn no es la cantidad de tools — un LLM puede llamar varias tools en una sola respuesta — sino la iteración: el modelo decide qué hacer entre vuelta y vuelta. "Y acá está la arquitectura que resuelve eso. Tomá una tarea como 'arreglá el bug en mi backend': una sola vuelta no alcanza, por más tools que llame el modelo en esa vuelta. Necesitás correr los tests, ver el output, decidir dónde mirar según el error, leer un archivo, proponer un cambio, correr los tests de nuevo. La decisión de qué hacer en cada paso depende de lo que se observó en el anterior. Esa arquitectura tiene un nombre — ReAct, de Reasoning + Acting entrelazados — y se ve así."

Después de cada observación, el modelo decide qué hacer a continuación. Esa decisión es lo que lo hace agente: no solo usa tools, las orquesta vuelta a vuelta según lo que va viendo."

Pensar — formular el próximo paso

Actuar — invocar herramientas

Observar — leer el resultado

Pensar

El agente razona sobre el estado actual: qué sabe, qué le falta saber, qué hipótesis tiene. No es pensamiento lineal — es el modelo formulando el próximo paso en función de lo que observó antes.

Produce: una decisión sobre qué acción tomar a continuación.

Actuar

El agente invoca una o varias herramientas: corre comandos, edita archivos, llama APIs, lee directorios. Las acciones producen un efecto en el entorno.

Produce: un cambio en el entorno — algo que antes no era verdad, ahora lo es.

Observar

El agente lee el resultado de la acción: el output del comando, el contenido del archivo, la respuesta HTTP. Eso nuevo entra a su contexto y alimenta la próxima vuelta de pensar.

Produce: información nueva en el contexto — el insumo de la siguiente vuelta.

"La primera fase se llama pensar. El agente recibe lo que hay en su contexto — el objetivo original, lo que observó en la vuelta anterior, la historia de lo que hizo — y razona sobre eso. No es lineal: no sigue un script. El modelo formula cuál es el próximo paso útil dado todo lo que sabe en ese momento. Qué hipótesis tiene, qué le falta ver, qué tiene sentido probar ahora. Lo que produce esta fase es una decisión: qué acción viene a continuación."

"La segunda fase es actuar. Y acá hay algo importante: actuar no significa 'escribir texto'. Significa invocar herramientas — una o varias en la misma vuelta. Correr un comando en la terminal, editar un archivo, hacer una llamada HTTP, leer un directorio. Las herramientas producen efectos en el entorno: algo que antes no era verdad, ahora lo es. El archivo tiene una línea nueva. El test corrió. El servidor respondió. Las tools en detalle las vemos en la siguiente sección — por ahora lo que importa es que actuar siempre toca algo concreto fuera del modelo."

"Tercera fase: observar. El agente lee el resultado de lo que acaba de hacer. El output del comando, el contenido del archivo, el código HTTP de la respuesta. Todo eso entra a su contexto. Y eso es clave: no es que el agente 'vio' el resultado y lo olvidó. Lo que observa queda en la ventana, disponible para el próximo pensar. Eso es lo que hace que cada vuelta del loop sepa más que la anterior. Y después repite — hasta que una condición de corte decide que terminó. ¿Qué se ve cuando el loop completo trabaja sobre un problema real?"

Condiciones de corte

Objetivo alcanzado

El estado del entorno coincide con lo pedido. Los tests pasan, el endpoint responde 200. Condición ideal.

Límite de pasos

Se alcanzó el techo configurado sin resolver el objetivo. El agente reporta qué hizo y hasta dónde llegó.

Pide ayuda

El agente no puede continuar sin input humano — falta un secreto, no tiene permiso, hay ambigüedad. El loop se suspende.

Falla no recuperable

El agente detecta un estado del que no sabe cómo salir. Para y reporta.

Tres ingenierías para dirigir un agente

Prompt engineering

Diseñar el mejor input al modelo: instrucciones, restricciones, ejemplos. Una sola vuelta de petición-respuesta.

Context engineering

alcance: una sesión

Controlar qué ve el modelo durante toda la tarea: qué se carga, qué se mantiene, qué se descarta. La ventana de contexto como recurso a gestionar.

Diseñar el entorno completo: qué tools tiene, qué guardrails lo frenan, qué scripts deterministas usa, qué feedback loops cierran el ciclo.

La respuesta del harness

Mecanismos que el entorno aporta para que esos modos no se manifiesten — o queden contenidos.

Guardrails

Gates antes de cada acción riesgosa: la tool no se ejecuta hasta que un humano (o una regla) autoriza. El modelo propone, el harness dispone.

Ej: permiso explícito antes de rm, antes de git push, antes de tocar producción.

Validación automática

Linters, type-checkers, tests, contratos. El modelo escribe, una verificación determinista chequea, y si falla se devuelve el error al loop.

Ej: cada edit corre tsc + tests; si rompe, el agente recibe el output y corrige.

Scripts deterministas

Operaciones críticas no se confían al modelo: se ejecutan por código. El agente puede pedir que corran, no puede improvisar la lógica.

Ej: el deploy es un script versionado; el agente lo invoca, no lo reinventa cada vez.

Feedback loops

Señales del entorno que vuelven al contexto cuando algo no anda. Sin observación, el agente no se autocorrige; con la observación correcta, sí.

Ej: CI roto inyecta el log de fallo; un endpoint 500 inyecta el stack trace.
