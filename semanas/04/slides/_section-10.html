Sintaxis: Tool(pattern)

Tool(pattern)
  • Bash(uv run pytest) — comando exacto; solo matchea esa línea sin argumentos.
  • Bash(uv run pytest *) — wildcard de argumentos; cubre uv run pytest tests/ -v y variantes.
  • Edit(./backend/**) — glob de path; cualquier archivo bajo backend/, recursivo.
  • Read(./.env) — archivo específico; útil en deny para impedir lectura de credenciales.
  • WebFetch(domain:fastapi.tiangolo.com) — dominio único; sintaxis especial para WebFetch.

Cuatro cosas que confunden de permisos

  • Bash(*) no es wildcard total. No existe una entrada que le dé permiso para correr cualquier comando Bash. El asterisco matchea argumentos de un comando específico — Bash(comando *), no Bash(*). Para autonomía amplia en Bash, hay que ser explícito comando por comando.
  • El default es preguntar, no denegar. Sin ninguna regla configurada, Claude pide confirmación antes de cada acción que toca el mundo. Eso suele ser lo que querés al arrancar — el propósito de allow es eliminar prompts donde ya decidiste dar autonomía.
  • Los wildcards no atraviesan ; ni &&. Si tenés Bash(uv run pytest *) en allow, eso no cubre uv run pytest && rm -rf .. El runtime valida el comando exacto contra el pattern; no podés colar comandos por concatenación.
  • Managed bloquea hacia abajo, sin escape. En un entorno corporativo, un admin puede fijar un deny en el nivel Managed que ningún proyecto ni usuario puede levantar. En proyectos personales no aplica; en equipos con políticas de seguridad, define el techo de lo posible.

Recorrer los cuatro gotchas. La uno se nombra rápido — es el error más común al empezar a configurar. La dos invierte una expectativa razonable (la gente espera que el default sea seguro = denegar). La tres es el dato de seguridad: no hay forma de pasar comandos extra por encadenación. La cuatro es el límite enterprise.

Reglas del runtime, deterministas, antes de cada tool call

Los permisos son reglas del runtime, evaluadas antes de cada tool call. Tres listas — allow / deny / ask — y una precedencia: deny gana sobre allow. Una regla escrita acá no se ignora ni se reinterpreta. Es el contrato más duro que tenés sobre lo que el agente puede hacer.

Los permisos limitan qué tools usa el agente. Las skills hacen lo contrario: le dan procedimientos que no traía. El agente sabe correr comandos — las skills le enseñan cuándo y en qué orden.

Una regla escrita acá no se ignora ni se reinterpreta. Es el contrato más duro que tenés sobre lo que el agente puede hacer."

"Y el bridge a §11. Toda esta sección fue sobre limitar — sobre acotar qué tools usa el agente. La próxima sección hace lo contrario: las skills le dan procedimientos que no traía. El agente ya sabe correr comandos; las skills le enseñan cuándo correrlos y en qué orden. Son las dos caras del mismo problema — qué puede hacer el agente — atacadas desde lados opuestos. Limitamos por un lado, extendemos por el otro."
