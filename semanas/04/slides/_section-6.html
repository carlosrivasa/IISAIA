Qué es Claude Code

  • Una CLI: la instalás una vez, la ejecutás como cualquier comando — claude en la terminal.
  • Abre una sesión interactiva con el modelo. Vos escribís, el modelo trabaja, te respondés en el mismo prompt.
  • Adentro de esa sesión corre el loop de §2 — pensar, actuar, observar, repetir — con tu repo como entorno real (no simulación).
  • Las tools que el modelo puede ejecutar son las del agente: Read, Edit, Write, Bash, Grep, Glob, WebFetch, entre otras.

Slide de definición directa. Qué es Claude Code en operación, sin metáforas. CLI + sesión interactiva + loop corriendo + tools concretas. Setup para los siguientes dos slides (sesión, y qué pasa al ejecutar claude).

"Claude Code es una CLI. La instalás una vez y la corrés desde la terminal escribiendo claude. Cuando la ejecutás, abre una sesión interactiva con el modelo: vos escribís, el modelo trabaja, te responde, y seguís en la misma sesión."

"Adentro de esa sesión, lo que corre es exactamente el loop que vimos en la parte 1. Pensar, actuar, observar, repetir. Con tu repo como entorno — entorno real, no simulación: cuando ejecuta tests, los tests corren de verdad; cuando edita un archivo, el archivo cambia."

"Las herramientas que tiene disponibles son las del agente: Read, Edit, Write, Bash para ejecutar comandos, Grep y Glob para buscar, WebFetch para traer documentación. Cada vez que el modelo decide actuar, elige una de esas. Las vamos a ver una y otra vez en todo lo que sigue."

Qué pasa cuando ejecutás claude

  • Camina hacia arriba desde el CWD recolectando los CLAUDE.md que encuentra — managed, user, project, local.
  • Lee .claude/settings.json (mergeado entre niveles): modelo, permisos, hooks, environment.
  • Carga las .claude/rules/ sin frontmatter (las que tienen paths: entran on-demand cuando matchean).
  • Carga el índice de skills y sub-agents disponibles (name + description), no el cuerpo.
  • Carga las primeras 200 líneas o 25 KB de MEMORY.md (auto memory).
  • Abre una ventana de contexto fresca con todo eso ya adentro, y queda esperando tu primer mensaje.

Todo esto está cargado antes de que escribas la primera tecla.

"Cuando escribís claude y dale enter, el runtime hace seis cosas antes de pedirte el primer mensaje."

"Uno: camina hacia arriba desde el directorio donde estás recolectando los CLAUDE.md que encuentra. Managed del sistema, user de tu home, project del repo, local si existe. Todos entran al contexto."

"Dos: lee el .claude/settings.json — mergeado entre niveles. De ahí saca qué modelo usar, qué permisos están configurados, qué hooks correr, qué environment variables inyectar."

"Tres: carga las rules de .claude/rules/. Las que no tienen frontmatter entran al arranque, igual que CLAUDE.md. Las que tienen paths declarados esperan: entran cuando el agente toca un archivo que matchee el glob."

"Cuatro: carga el índice de skills y sub-agents disponibles. Ojo: solo el name y description, no el cuerpo de cada uno. El cuerpo entra cuando se invoca."

"Cinco: carga las primeras doscientas líneas o veinticinco kilobytes del MEMORY.md — la auto memory que Claude se escribe solo entre sesiones."

"Y seis: con todo eso en la ventana, queda esperando tu primer mensaje. Lo que importa retener: todo lo que enumeré ya está cargado antes de que vos escribas una letra. La sesión arranca con una ventana de contexto que ya tiene contenido."

Qué es una sesión

  • Una conversación con su propia ventana de contexto. Todo lo que se acumula durante la sesión (tu input, las respuestas del modelo, los resultados de tools) vive ahí.
  • Mientras la terminal esté abierta, la sesión mantiene el historial. Vos seguís haciendo tareas y el modelo recuerda lo anterior.
  • Al cerrar la sesión (Ctrl+C o cerrar terminal), la ventana desaparece. La próxima vez que ejecutes claude, arrancás con una ventana nueva.
  • Lo único que sobrevive entre sesiones es la auto memory (MEMORY.md) — lo que Claude consideró que valía la pena guardar.
  • Podés correr múltiples sesiones en paralelo (varias terminales). Cada una con su ventana propia, independientes entre sí.

Recorrer los cinco bullets. Marcar fuerte "la ventana desaparece" en el bullet 3 — es la confusión más frecuente.

Slide de definición de sesión. Cinco propiedades operativas. La que más confunde es la mortalidad de la ventana: la gente cree que Claude "recuerda" entre sesiones y arma estrategias asumiendo eso. La auto memory es la única excepción, y es acotada.

"Una sesión es una conversación con su propia ventana de contexto. Todo lo que pasa dentro de la sesión — tu input, las respuestas del modelo, los outputs de los comandos que ejecutó — vive en esa ventana."

"Mientras la terminal esté abierta, la sesión mantiene el historial. Podés trabajar en cosas que se conectan: pedirle algo, ver el resultado, pedirle algo más basado en lo anterior. El modelo se acuerda porque todo está en la ventana."

"Cuando cerrás la sesión — Ctrl+C, cerrar terminal, lo que sea — esa ventana desaparece. La próxima vez que abrís claude, arrancás con una ventana nueva. El modelo no se acuerda de la sesión anterior."

"Lo único que sobrevive entre sesiones es la auto memory, el MEMORY.md que vimos al principio. Pero eso lo escribe el propio Claude cuando decide que algo vale la pena recordar — no es el historial entero."

"Y un detalle práctico: podés correr varias sesiones en paralelo, en distintas terminales. Cada una tiene su propia ventana, independientes. Para tareas paralelas o experimentos en ramas distintas, abrís otra terminal y listo."
