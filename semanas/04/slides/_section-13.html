Uno de los cuatro permission modes — el dial de autonomía

Mode Cuánta autonomía da
default Pregunta antes de cualquier acción sin regla allow. El modo del día a día.
acceptEdits Auto-acepta Edit y Write. Sigue preguntando para Bash.
plan Solo lectura hasta que aprobás un plan con ExitPlanMode. No escribe, no ejecuta.

Tres vías para activar plan mode

  • Flag al arranque — cuando ya sabés que la tarea pide la pausa antes de empezar. claude --permission-mode plan
  • Campo en settings.json — para fijar el mode por default en este proyecto o para tu usuario. "permissionMode": "plan"
  • Shift+Tab durante la sesión — cicla entre los cuatro modes sin reiniciar. La UI muestra el mode activo en el banner.

Regla de fondo: cuanto menos reversible es la tarea, más vale entrar al proceso antes de la primera acción.

Tres vías en orden — la primera y la segunda son configuración, la tercera es operación en vivo. La tercera es la más útil en la práctica: empezás en default, ves que el agente está por hacer algo costoso, Shift+Tab a plan, sin perder la sesión. La nota final es la regla de oro: la reversibilidad de la tarea define cuándo querés la pausa.

El flujo del plan — seis pasos, un contrato

  1. Activar plan mode — cualquiera de las tres vías.
  2. Describir la tarea al agente, como siempre.
  3. Claude propone un plan: qué archivos va a tocar, en qué orden, qué tools va a usar.
  4. Revisar. Corregir si hace falta — "no toques tests/", "empezá por el módulo X".
  5. Aprobar con ExitPlanMode.
  6. Claude ejecuta la versión aprobada.

El plan es el contrato. Una vez aprobado, Claude ejecuta sin volver a parar en cada acción — salvo que un deny del bloque de §10 lo detenga.

Tres cosas que confunden de plan mode

  • Plan mode no es sandbox. Una vez aprobado el plan, Claude ejecuta — incluyendo Bash con consecuencias reales. El gate es de proceso (cuándo empieza la ejecución), no de capacidad (qué puede hacer mientras ejecuta).
  • Modes y permisos son ortogonales. El mode controla cuándo arranca la ejecución; el deny de §10 controla qué tools puede usar durante la ejecución. Los dos juntos = control fino. Cada uno por separado es parcial.
  • Plan mode protege contra deriva, no contra mal código. Si el plan es bueno pero el código que escribe el agente tiene un bug, el mode no lo va a atajar — para eso están los tests, el code review, y el ojo del supervisor sobre el diff final.

Recorrer los tres bullets. El primero desarma la confusión más frecuente — la gente asume que plan mode es un modo seguro de ejecutar, no un modo de pausar antes de ejecutar. El segundo separa explícitamente dos sistemas que la gente tiende a confundir. El tercero acota qué problema resuelve y cuál no — plan mode no es una panacea, es una pieza específica del runtime.

"Quinta pregunta. Qué confunde. Tres cosas, las más frecuentes."

"Una. Plan mode no es sandbox. Esta es la confusión más común. La intuición natural es 'si está en plan mode, está protegido contra errores'. No. Plan mode pone un gate antes de que empiece la ejecución — aprobás el plan, el agente ejecuta. Y lo que ejecuta puede incluir comandos Bash con consecuencias reales: rm, git push, npm install, lo que sea. El gate es de proceso, cuándo empieza la ejecución, no de capacidad, qué puede hacer mientras ejecuta. Si querés limitar la capacidad, eso vive en deny — la sección que ya vimos."

"Dos. Modes y permisos son ortogonales. Y este es el punto fuerte de la sección. Los dos sistemas — los permission modes que vimos acá y los permisos allow/deny/ask de §10 — son ejes distintos. El mode controla cuándo arranca la ejecución. Los permisos controlan qué tools puede usar mientras ejecuta. Los dos juntos te dan el control fino. Cada uno por separado es parcial: plan mode sin deny te deja vulnerable durante la ejecución; deny sin plan mode te deja sin la oportunidad de revisar antes de que arranque."

"Y tres. Plan mode protege contra deriva, no contra mal código. Si el plan es bueno — los archivos correctos, los pasos correctos, el orden correcto — pero el código que escribe el agente cuando ejecuta tiene un bug, el mode no lo va a atajar. Plan mode resuelve un problema específico: que el agente no entendió la tarea y empieza a hacer algo distinto de lo que pediste. Para detectar bugs en el código que sí entendió bien, están los tests, el code review, y el ojo del supervisor sobre el diff final. Plan mode no es panacea — es una pieza del runtime que resuelve una cosa, no todas."
