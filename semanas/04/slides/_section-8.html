Un archivo .md por tema, organizable y opcionalmente condicional

  • Archivo Markdown corto en .claude/rules/ sobre un tema acotado — convenciones de código, reglas de seguridad, criterios de testing, convenciones de API. Un archivo por tema.
  • Sin frontmatter: se carga al arranque como un fragmento de CLAUDE.md. Ocupa contexto siempre.
  • Con frontmatter paths:: path-scoped — entra al contexto solo cuando el agente toca un archivo que matchea alguno de los globs declarados.
  • → La diferencia operativa con CLAUDE.md es modularidad por archivo + scope opcional por path, no la sintaxis.

Dos ubicaciones, descubrimiento recursivo

Nivel 1 Usuario
Reglas personales — todos tus proyectos ~/.claude/rules/ Convenciones tuyas que aplicás siempre. No se versionan con ningún repo.
Nivel 2 Proyecto
Reglas del equipo — versionadas con el código ./.claude/rules/ Convenciones del repo: código, tests, seguridad, APIs. Las leen todos los que abren el proyecto.

Podés agrupar rules en subcarpetas como rules/backend/ o rules/deploy/: Claude Code recorre .claude/rules/ entera al cargar, no hace falta listar los archivos.
La carga la decide el frontmatter del archivo, no la subcarpeta donde vive — agrupar es sólo organización.

Las subcarpetas son sólo organización; la condicionalidad la da el frontmatter."

El frontmatter decide cuándo entra la rule al contexto

Modo A

Sin frontmatter  — siempre cargada

# code-style.md
- Líneas <100 caracteres.
- Funciones <50 líneas.
- Imports: stdlib → terceros → proyecto.

El archivo arranca directamente con el cuerpo, sin bloque YAML adelante. El runtime la carga al iniciar la sesión, igual que un CLAUDE.md.

Costo: mismo que CLAUDE.md.

Modo B

Con paths:  — path-scoped

---
paths:
  - "backend/**/*.py"
---

# API conventions ...

El frontmatter declara uno o más globs. La rule entra al contexto solo cuando el agente lee, edita o escribe un archivo cuyo path matchea.

Costo: cero hasta que es relevante.

Frontmatter YAML arriba, cuerpo Markdown abajo

.claude/rules/api.md (path-scoped al backend)
---
paths:
  - "backend/**/*.py"
---

# API conventions

Cuando trabajes con código del backend:

- Cada endpoint declara su `response_model` y `status_code` explícito.
- Para escritura (POST/PUT/DELETE), agregar `dependencies=[Depends(require_bearer)]`.
- Si tocás o agregás un endpoint, regenerar `openapi.yaml` con
  `uv run python scripts/export_openapi.py` y commitearlo en el mismo PR.

Globs soportados

**/*.py cualquier .py del árbol, a cualquier profundidad src/**/* todo lo que cuelga de src/ *.md Markdown solo en la raíz del proyecto backend/**/*.{py,yaml} expansión de llaves para dos extensiones

Las cuatro rules del demo-repo

  • code-style.md  — sin frontmatter, siempre cargada. Convenciones de Python: largo de línea, imports, logging.
  • testing.md  — sin frontmatter, siempre cargada. Test client de FastAPI, naming, fixture reset_store.
  • security.md  — sin frontmatter, siempre cargada. Manejo del bearer token y del .env; no loggear secretos.
  • api.md  — paths: ["backend/**/*.py"], solo al tocar backend/*.py. response_model explícito, auth en escritura, regenerar openapi.yaml.

Tres cosas que confunden de Rules

Caso 1

No es .cursorrules

Cursor tiene su propio sistema con ese nombre y no es compatible con Claude Code. Si encontrás tutoriales que mezclan .cursorrules con Claude Code, están confundiendo dos herramientas distintas. El mecanismo es independiente.

Caso 2

Sin paths:, cuesta lo mismo que CLAUDE.md

Partir las convenciones en code-style.md, testing.md, security.md mejora la organización del repo — no baja el costo en tokens. Para reducir contexto necesitás declarar paths:. La modularidad es organizativa por default; condicional solo con frontmatter.

Caso 3

El trigger es leer el archivo, no el path

Listar un directorio o correr un comando de shell que mencione un path del backend no activa una rule scoped. El trigger es acceder al contenido — Read, Edit o Write — de un archivo cuyo path matchea el glob. Incluso un Read alcanza.

Recorrer las tres cards de izquierda a derecha. En la dos, dejar caer la frase "modularidad organizativa por default, condicional solo con frontmatter". Es la línea que más se confunde.

"Quinta y última pregunta. Qué confunde. Tres cosas, todas las he visto en clase."

"Una. No es punto cursorrules. Cursor tiene su propio mecanismo de reglas que se llama así, y no es compatible. Si googleás 'claude code rules' y te aparecen tutoriales mezclados, fijate que no estén hablando de Cursor — son dos productos distintos con sistemas distintos."

"Dos. Y esta es la que más confunde: sin paths, una rule cuesta exactamente lo mismo que CLAUDE.md. Mucha gente parte el CLAUDE.md en tres rules pensando que achicó la ventana — la ventana queda igual. Lo que ganaste es que el repo está mejor organizado por tema. La modularidad de rules es organizativa por default; se vuelve condicional solo cuando agregás paths."

"Tres. El trigger es leer el archivo, no que el path aparezca. Si Claude lista el directorio backend con un ls, eso no activa api.md. Si corre un grep que produce líneas con paths del backend, tampoco. La rule se activa cuando alguna de las tres tools — Read, Edit, Write — abre el contenido de un archivo cuyo path cae en el glob. Incluso un Read solo alcanza para que entre."

Modulares por archivo, condicionales por path

Rules es la disciplina de partir las instrucciones del agente en archivos chicos por tema, con la opción de declarar cuándo cada una entra al contexto — no solo qué dice. Modulares por archivo, opcionalmente condicionales por path.

Hasta acá, todo lo que vimos son instrucciones para el agente: Markdown que le dice qué hacer.
Lo que viene es distinto en naturaleza — configuración para el runtime: JSON que le dice al programa que lo envuelve cómo correrlo.

No es 'cuando trabajes en backend, hacé esto'; es 'cuando Claude termine una sesión, ejecutá este hook'. Cambia el canal: del agente al sistema que lo corre."
