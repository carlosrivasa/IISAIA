Dos objetos distintos — un procedimiento y su disparador

  • Skill — procedimiento completo empaquetado como objeto de conocimiento: fases numeradas, checkpoints de verificación, lista explícita de anti-patterns. No es una instrucción suelta ni una preferencia de estilo: describe exhaustivamente cómo se hace un tipo de tarea.
  • Slash command — atajo. Puede invocar una skill, ejecutar instrucciones inline, o disparar un comportamiento built-in del runtime. Su cuerpo es mínimo por diseño.
  • Skill ≠ slash command. Una skill puede existir sin slash command (se activa por relevancia). Un slash command puede existir sin skill (instrucciones inline, atajos a features del sistema).
  • → La skill es el conocimiento. El slash command es el disparador.

Tres ítems con pausa entre cada uno. El tercero es el que más confunde — repetir "skill ≠ slash command" para fijarlo. Pausa antes del callout. El callout es la línea a retener.

Al arranque entra el índice, no el procedimiento

De cada skill, al arranque entra solo name + description.
El cuerpo — fases, checklists, anti-patterns — entra cuando se invoca.

  • En /context las vas a ver listadas — con ~60–110 tokens cada una. Eso es el índice (name + description) que Claude necesita para decidir si activarla por relevancia.
  • El SKILL.md completo (varios cientos o miles de tokens) entra a la ventana al invocarse explícitamente o al matchear el description con la tarea actual.
  • Slash commands aparecen en el autocomplete cuando escribís /. Tampoco cargan su cuerpo hasta invocación.

Diseño, no limitación. Tenés cinco skills instaladas: pagás cinco descripciones cortas siempre, y el cuerpo de la única que usás esa sesión. Las otras cuatro no se expanden.

Anatomía del SKILL.md — add-endpoint del demo-repo

---
name: add-endpoint
description: Use when the user asks to add a new REST endpoint to the demo-repo backend. Adds the route, the schema if missing, the test, and regenerates openapi.yaml.
---
  1. Decidir verbo y path — confirmar con el usuario; escrituras requieren Depends(require_bearer).
  2. Schemas — si hace falta, agregar modelos en backend/schemas/models.py.
  3. Router — función en backend/routers/<resource>.py con response_model y status_code explícitos.
  4. Test — happy path + error esperado en tests/test_<resource>.py.
  5. OpenAPI — uv run python scripts/export_openapi.py.
  6. Verificación — uv run pytest -v pasa entero; mostrar diff del openapi.yaml.

Anti-patterns: crear el endpoint sin test · olvidar regenerar openapi.yaml · dejarlo sin response_model · hardcodear constantes que pertenecen a models.py o a una env var.

Tres cosas que confunden de skills y slash commands

  • No todo slash command invoca una skill. /memory, /context, /permissions son atajos a comportamientos built-in del runtime. No hay skill detrás de ellos. El mecanismo es el mismo — un archivo en .claude/commands/ — pero el cuerpo ejecuta instrucciones directas o llama features del sistema.
  • El description es contrato de activación, no estilo. Cuando Claude decide si una skill aplica, lee ese campo. Vago activa mal — la skill entra cuando no corresponde, o no entra cuando sí. La convención Use when X no es preferencia estética: es la forma del contrato.
  • El flow de ejecución es lineal. Cuando Claude entra en una skill, lee SKILL.md completo y ejecuta las fases en orden. Seis fases, las seis. No saltea, no reordena, no improvisa dentro del procedimiento. Esa garantía es la razón de usar una skill en lugar de escribir un prompt libre.

Recorrer los tres bullets. El primero desarma una falsa equivalencia común. El segundo es operativo — si escribís skills propias, este es el punto que más impacta su utilidad. El tercero es la propiedad estructural — la razón por la que vale la pena empaquetar un procedimiento como skill.

Skills aíslan un procedimiento — entran cuando se necesitan

Las skills aíslan un procedimiento: fases, checkpoints, anti-patterns, todo empaquetado. Entran al contexto cuando se invocan, no antes. Lo que el agente no usó no paga tokens. Y cuando entran, el flow es lineal — fases en orden, sin improvisación.

Las skills aíslan un procedimiento. Los sub-agents aíslan un loop entero — el mismo principio (aislar para proteger el contexto del padre) aplicado a una unidad más grande.
