Leer

reversible

La asimetría reversible/no reversible es la motivación operativa del harness. Sin esta distinción, todas las tools parecerían iguales — con ella se justifica que algunas necesitan guardrails extra.

"Antes de ver los modos de falla y la respuesta del harness, una distinción que organiza todo lo que sigue: no todas las tools tienen el mismo peso cuando algo sale mal."

"Una tool que lee no cambia el estado del entorno. Si el agente leyó el archivo equivocado, lo peor que pasó es que llenó el contexto con ruido. Puede intentar otra lectura, retroceder, corregir."

"Una tool que escribe o ejecuta modifica el entorno. Un archivo sobreescrito sin backup. Un git commit que avanzó el historial. Un endpoint que procesó algo. Puede que sea recuperable, puede que no. No hay deshacer automático."

"Por eso el harness mira más las tools que actúan sobre el mundo. Donde el daño es reversible, el costo de equivocarse es tiempo; donde no lo es, el costo es el estado del entorno. Esta asimetría es lo que justifica los guardrails que vienen."

La respuesta del harness

Mecanismos que el entorno aporta para que esos modos no se manifiesten — o queden contenidos.

Guardrails

Gates antes de cada acción riesgosa: la tool no se ejecuta hasta que un humano (o una regla) autoriza. El modelo propone, el harness dispone.

Ej: permiso explícito antes de rm, antes de git push, antes de tocar producción.

Validación automática

Linters, type-checkers, tests, contratos. El modelo escribe, una verificación determinista chequea, y si falla se devuelve el error al loop.

Ej: cada edit corre tsc + tests; si rompe, el agente recibe el output y corrige.

Scripts deterministas

Operaciones críticas no se confían al modelo: se ejecutan por código. El agente puede pedir que corran, no puede improvisar la lógica.

Ej: el deploy es un script versionado; el agente lo invoca, no lo reinventa cada vez.

Feedback loops

Señales del entorno que vuelven al contexto cuando algo no anda. Sin observación, el agente no se autocorrige; con la observación correcta, sí.

Ej: CI roto inyecta el log de fallo; un endpoint 500 inyecta el stack trace.

Las cuatro respuestas del harness contra los modos de falla. Guardrails frenan acciones riesgosas; validación detecta errores antes de propagarlos; scripts deterministas evitan que el modelo improvise donde no debe; feedback loops aseguran que el agente vea cuando algo falló.

"Para cada modo de falla, el harness tiene una respuesta. Cuatro mecanismos que diseñás como parte del entorno — no son cosas que le pedís al modelo, son cosas que el entorno hace por su cuenta."

"Guardrails: gates antes de cada acción riesgosa. La tool no se ejecuta hasta que se autoriza — un humano clickea, o una regla del harness lo permite. El modelo propone la acción; el harness decide si pasa. Esto neutraliza la mayoría de los daños no reversibles."

"Validación automática: linters, type-checkers, tests, contratos. Después de cada edit, una verificación determinista corre. Si falla, el output va al contexto y el loop sigue corrigiendo. Es la diferencia entre 'el modelo dice que funciona' y 'la máquina lo verifica'."

"Scripts deterministas: las operaciones críticas — deploy, migración, llamadas a servicios externos — no se confían al modelo. Se ejecutan por código versionado. El agente puede pedir que corran, no puede improvisar la lógica cada vez. Esto neutraliza la deriva en partes sensibles."

"Feedback loops: el entorno le devuelve señales al agente cuando algo no anda. Tests rotos, builds fallidos, endpoints que devuelven 500. Sin esta retroalimentación, el agente no se autocorrige. Con ella, el loop converge."

"La auditoría sigue siendo necesaria — uno revisa lo que el agente hizo, no le cree a ciegas. Pero la supervisión real, la que evita daños, vive en el harness, no en la revisión final."

Tres niveles para dirigir un agente

Prompt engineering

una interacción

Diseñar el mejor input al modelo. Lo que vienen practicando desde antes.

Context engineering

una sesión

Controlar qué ve el modelo: cargar lo necesario, mantener la señal, compactar o resetear cuando se llena.

Harness engineering

múltiples sesiones

Diseñar el entorno: tools, guardrails, scripts deterministas, feedback loops.

Cada feature de Claude Code que viene llena uno de estos tres niveles.

Cuando veas una feature nueva, la pregunta ya no es "¿qué hace?" — es "¿en qué nivel actúa?"
