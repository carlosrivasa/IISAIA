Las dimensiones del harness

Diseñar un harness es decidir, simultáneamente:

Estructurás el prompt

Qué instrucciones de sistema le das, qué prioridades, qué tono. Lo que el modelo lee antes de hacer nada.

Exponés tools

Qué puede invocar el agente y qué no. El menú de manos a disposición — y lo que dejás afuera.

Cargás contexto

Qué información tiene a mano cuando arranca y vuelta tras vuelta. Acá la frontera con context engineering se difumina.

Armás feedback loops

Cómo el entorno le devuelve señal cuando algo no anda — tests rotos, logs, errores de tipo, builds fallidos.

Verificás outputs

Linters, type-checkers, tests automáticos, contratos. Que no confíes ciego en lo que el modelo dice que hizo.

Descomponés tareas

Cuándo el agente delega a un sub-agente, cuándo invoca un script determinista, cuándo te pide ayuda.

La frontera con context engineering

Los límites son difusos. "Qué contexto cargás" es parte del harness y es lo que define context engineering. Hay solapamiento.

Algunos usan "harness" específicamente para el andamiaje de runtime: el system prompt, las definiciones de las tools, la lógica de reintentos, la orquestación de sub-agentes, los pasos de verificación.

Eso, en esa lectura, es distinto del modelo en sí y de un prompt suelto. Es la capa de scaffolding que envuelve a los dos.

Tres niveles para dirigir un agente

Prompt engineering

una interacción

Diseñar el mejor input al modelo. Lo que vienen practicando desde antes.

Context engineering

una sesión

Controlar qué ve el modelo: cargar lo necesario, mantener la señal, compactar o resetear cuando se llena.

Harness engineering

múltiples sesiones

El andamiaje: prompt, tools, contexto, feedback, verificación, descomposición.

Cada feature de Claude Code que viene llena uno de estos tres niveles.

Cuando veas una feature nueva, la pregunta ya no es "¿qué hace?" — es "¿en qué nivel actúa?"
