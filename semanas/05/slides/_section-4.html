El problema que ataca

Sin metodología

Le pedís algo → el agente arranca a tipear código → el desfasaje entre tu intención y su interpretación aparece tarde, con 20 mensajes y archivos commiteados.

Con metodología

Entendé qué construís → escribí el spec → planificá → ejecutá → verificá → mergeá.

Detectar el desfasaje en palabras es barato. En código ya escrito es caro.

Diagnóstico del problema. Comparación sin/con. El cierre martilla el costo del desfasaje tardío.

"El problema sin metodología es conocido. Le pedís al agente que construya algo y arranca a tipear código antes de saber qué estás construyendo. Si hay un desfasaje entre lo que vos tenés en la cabeza y lo que él entendió, el desfasaje aparece tarde — cuando ya hay archivos, tests, commits, y la conversación lleva veinte mensajes."

"Lo que cambia con metodología es la secuencia: entendé qué construís, diseñalo, planificá, ejecutá, verificá, mergeá. Cada paso tiene un punto de control antes de pasar al siguiente."

"La razón por la que esto importa: detectar desfasaje en palabras es barato. Detectarlo en código ya escrito es caro. Esa es toda la apuesta de Superpowers."

"Brainstorming: convierte una idea cruda en un design spec. El spec termina en disco como archivo, no como conversación perdida."

"Writing-plans: toma ese spec aprobado y lo convierte en un plan de tareas chicas, también en disco. Spec más plan: el contrato escrito."

"GitHub Flow envuelve todo lo que sigue. No es una skill — es la convención de git: abrís una branch antes de tocar código, y cuando todo está listo, abrís un PR. El plugin se lleva bien con este patrón."

"Subagent-driven-development: ejecuta el plan despachando un subagent fresco por cada tarea. Cada tarea arranca con contexto limpio."

"Test-driven-development: dentro de cada tarea, el subagent arranca con un test que falla. Rojo, verde, refactor. Sin TDD, 'ya funciona' termina siendo 'compila'."

"Requesting-code-review: cuando el plan está implementado, otro agente — distinto del que implementó — revisa el diff contra el spec. Strengths e issues clasificados por severidad."

"Verification-before-completion: cuando el agente cree que terminó, antes de decírtelo, vuelve a correr los tests y los comandos del plan y pega la salida en el mensaje de cierre. Si no hay salida pegada, no puede declarar 'listo'. Existe porque los modelos suelen afirmar 'los tests pasan' sin haber ejecutado nada — y eso te lo lleva descubrir mañana cuando algo se rompe en producción."

"Finishing-a-development-branch: cuando todo está verificado, te muestra el menú de opciones — PR, merge, branch viva con plan, descartar. Cierra el ciclo. Este mapa se va a iluminar caja por caja en las próximas secciones. Quédense con el mapa."

Cuatro principios, declarados en el README

Test-Driven Development

Tests primero, siempre. Un test escrito después del código valida lo que el código hace, no lo que vos querías que hiciera.

Sistemático antes que improvisado

Un proceso definido por encima de improvisar. Si no hay un proceso, lo estás inventando cada vez.

Complexity reduction

Simplicidad como objetivo primario. El código más simple que cumple es el correcto.

Evidence over claims

Verificar antes de declarar éxito. «Los tests pasan» sin el output al lado es una afirmación, no un hecho.

Cuatro principios filosóficos del plugin. Ninguno es nuevo; lo nuevo es aplicarlos al loop con el agente.

"Cuatro principios, declarados explícitamente en el README del plugin."

"Test-Driven Development: escribir tests primero, siempre. La razón es práctica — un test escrito después del código valida lo que el código hace, no lo que vos querías que hiciera. Pierde su valor como red de seguridad."

"Sistemático antes que improvisado: un proceso definido por encima de improvisar. Las skills son la implementación de esa idea — no es que el plugin tiene 'tips para hacer mejor código', es que tiene UN proceso definido para cada decisión."

"Complexity reduction: simplicidad como objetivo primario. Cualquier feature que se pueda hacer con menos código, se hace con menos código."

"Evidence over claims: verificar antes de declarar éxito. La diferencia entre 'creo que funciona' y 'acá está el comando y su salida'. Lo retomamos en §12."

"Ninguno de los cuatro es nuevo en ingeniería de software. Lo nuevo es que el plugin los aplica al loop con el agente, no sólo al código que el agente produce."
