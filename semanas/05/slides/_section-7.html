Antes de writing-plans, el marco

Si cambia el spec, cambia el plan. Si cambia el plan, cambia el código.
Nunca al revés.

El código no le dicta nada al spec. Si en cualquier paso descubrís que la entrada está mal, subís al paso anterior y lo arreglás ahí.

Slide-marco: la regla unidireccional de spec-driven development. Es el contrato conceptual de toda la cadena.

"Antes de meternos en qué hace writing-plans, vale tener claro el marco que la skill obedece. Es una regla simple: si cambia el spec, cambia el plan. Si cambia el plan, cambia el código. Nunca al revés."

"En la práctica significa: si en pleno coding descubrís que el plan está mal, no compensás en el código — volvés al plan y lo arreglás ahí. Si el plan está mal porque el spec estaba mal, volvés al spec. La dirección es siempre desde lo abstracto hacia lo concreto."

Qué hace y cuándo se activa

Qué hace

Toma el spec aprobado y lo convierte en un plan markdown con tareas chicas — de 2 a 5 minutos cada una, con paths exactos, código completo en cada step, y comandos de verificación al lado.

Asunción agresiva: el ingeniero que va a ejecutar no conoce el codebase ni el dominio.

Cuándo se activa

Sí se activa

Después de brainstorming, antes de tocar código. El spec aprobado es la entrada.

No, sin spec

Si pedís plan para una tarea grande sin pasar por brainstorming, lo primero que hace es recomendar volver atrás y producir el spec.

Por qué importa y dónde no se cede

Por qué importa

El plan en disco es lo que permite que cualquier agente — vos mañana, un subagente, un colega que toma la posta — ejecute la implementación sin re-deducir nada.

Sin esta skill, el plan vive en tu cabeza y se pierde cuando se cierra la sesión.

Punto crítico

Tareas de 2 a 5 minutos con archivos exactos y código completo en cada step. Nada de «similar a la Task N» sin repetir el código. Nada de «add appropriate error handling» sin mostrar qué.

Razón: el ejecutor lee tareas fuera de orden, y un subagente arranca cada task con contexto fresco.

Detalles del flujo

Dos cosas importantes

El plan en disco

El plan se persiste en:

docs/superpowers/plans/
  YYYY-MM-DD-<feature>.md

Se commitea junto al spec. Ese par es lo que subagent-driven-development lee para ejecutar autónomamente.

Plan Self-Review

Antes de pasártelo, la skill chequea sola:

  • Spec coverage: ¿hay tarea por cada requisito del spec?
  • Placeholders: TBD, TODO, «implement later»
  • Type consistency: clearLayers() en Task 3 no puede ser clearFullLayers() en Task 7

Tres anti-patrones a evitar

TBD / TODO / «implement later» son fallas del plan, no pendientes legítimos. El plan se completa o se reduce su scope.
«Add appropriate error handling» sin mostrar qué error handling. Vago = no implementable.
«Similar a la Task N» sin repetir el código. Repetí el código en cada step que lo necesita.

Dos reveals — uno por sub-sección.

Slide doble: dos detalles operativos + tres anti-patrones a evitar.

"Dos cosas importantes. La primera: el plan se persiste en docs/superpowers/plans con el mismo patrón fecha + topic que vimos en el spec. Ese par — spec más plan — es lo que después lee subagent-driven-development para ejecutar autónomamente. Si querés cambiar el rumbo del feature, editás esos archivos y volvés a correr el flujo. La segunda: hay un Plan Self-Review interno con tres chequeos: spec coverage — ¿hay tarea por cada requisito del spec?; placeholder scan — TBD, TODO, implement later son red flags; type consistency — si en Task 3 llamaste a algo clearLayers, en Task 7 no puede aparecer como clearFullLayers."

"Y tres anti-patrones del plan en sí. TBD, TODO, 'implement later' son fallas del plan, no pendientes legítimos — el plan se completa o se reduce su scope. 'Add appropriate error handling' sin mostrar qué error handling concreto: vago equivale a no implementable. 'Similar to Task N' sin repetir el código: la gente lee tareas fuera de orden, y un subagente arranca cada task con contexto fresco — repetí el código en cada step que lo necesita."
