requesting-code-review

Una pasada con ojos frescos antes de claimar listo.

Opener de §11. Quinta skill del happy-path; pieza s11 iluminada en el roadmap.

"Quinta skill: requesting-code-review. Cuando subagent-driven-development cierra todas las tareas con tests verdes, todavía falta una pasada — un review estructurado contra el plan, con un agente distinto del que implementó. Eso es lo que hace esta skill."

Implementer y reviewer son agentes distintos

Implementer

El que escribió el código

  • plan y spec
  • toda la sesión de implementación
  • las decisiones tácitas que tomó
→

Reviewer

Agente fresco, dispatchado aparte

  • el diff
  • el spec original

no ve el historial de sesión

Sin el historial, el reviewer no carga el sesgo del autor ni la narrativa de «ya lo discutimos y quedó así».

El concepto central de la skill: el reviewer es un agente separado y limitado, no la continuación del implementer.

"El concepto central es éste: el implementer y el reviewer son agentes distintos. El implementer es el que escribió el código — vio el plan, el spec, toda la sesión, las decisiones intermedias. El reviewer es un agente fresco, dispatchado aparte, que sólo ve el diff que se generó y el spec original. No ve el historial de sesión."

"Eso es a propósito. Sin el historial, el reviewer no carga el sesgo de quien implementó ni la narrativa de 'ya lo discutimos y quedó así'. Llega como un par de ojos limpios al código terminado."

Cuándo se activa y por qué importa

Cuándo se activa

Al final, antes de cerrar

Cuando terminás de implementar el plan y los tests están verdes. También antes de mergear cambios que tocan más de un par de archivos.

Útil también si estás trabado

Un agente fresco lee el trabajo y marca lo que vos ya naturalizaste y no ves.

Por qué importa

Code review encuentra cosas distintas a las que encuentran los tests: dead code, mismatch entre spec y código, ergonomía pésima de API pública, archivos que crecieron demasiado, convenciones del proyecto rotas sin que los tests fallen.

Tests verifican comportamiento. Review verifica diseño.

Dos reveals — cuándo + por qué.

Cuándo se invoca la skill + qué encuentra esta pasada que no encuentran los tests.

"Cuándo se activa: cuando terminás de implementar el plan, después de que TDD pasó verde, antes de pasar a finishing-branch. También antes de mergear features grandes o cualquier cambio que toque más de un par de archivos. Y un caso menos obvio: si estás trabado y no sabés qué falta, también sirve — un agente fresco lee el trabajo y te marca lo que vos ya no ves."

"Por qué importa: code review encuentra cosas distintas a las que encuentran los tests. Dead code que quedó de una iteración anterior. Mismatch entre lo que dice el spec y lo que hace el código. Ergonomía pésima de la API pública. Archivos que crecieron demasiado. Convenciones no escritas del proyecto — por ejemplo cómo se manejan los errores o cómo se nombran los endpoints — que el código viola sin que los tests se enteren. Los tests verifican comportamiento; el review verifica diseño. Son dos cosas distintas y se complementan."

Tres severidades

Crítico

Se resuelve YA. Bloquea el avance.

No se «anota para después».

Mayor

Aprobación humana antes de mergear.

No degradarlo a menor para no atrasar.

Menor

Nota en el PR como contexto.

No requiere acción inmediata.

Las tres severidades + qué hacer con cada una. La severidad la decide el reviewer y es informativa, no negociable.

"El reviewer devuelve un report con dos partes: strengths — qué quedó bien — e issues clasificados por severidad. Los issues vienen en tres niveles."

"Críticos: se resuelven YA. Bloquean el avance. No se 'anotan para después' — se arreglan en el momento. Si el spec cambió durante la implementación, se modifica el plan; si el spec se mantiene, se ajusta el código."

"Mayores: requieren aprobación humana explícita antes de mergear. No los degradás a menores 'para no atrasar' — si el reviewer marcó algo como mayor, hay una razón; pedile que la explique antes de bajarlo de nivel."

"Menores: notas que pueden quedar en el PR como contexto. No requieren acción inmediata, pero quedan registrados para que el próximo que toque el código sepa que están ahí."

Reviewer final del demo-repo

Después de las 11 tasks, el coordinador dispatcheó un reviewer fresco sobre el branch entero. Resultado: READY con 3 violaciones de reglas del proyecto.

run_import() tiene 71 líneas

Regla del proyecto (CLAUDE.md): funciones < 50 líneas.

Router POST sin status_code= explícito

Regla de api.md: cada endpoint debe declarar su status code.

admin.js:33 tiene 103 caracteres

Regla del proyecto: línea < 100 caracteres.

Tratados como blockers, arreglados antes de cerrar. Commit 4988512 — branch verificado.

Caso real del log. El reviewer final encontró 3 issues concretos referenciando las propias reglas del proyecto. Datos extraídos del log de la sesión.

"Acá lo vemos en la práctica. Cuando subagent-driven cerró las 11 tasks, el coordinador dispatcheó un reviewer fresco para que mirara el branch entero. El verdict overall fue READY — el código pasa los tests y cumple el spec — pero igual encontró tres violaciones de reglas del proyecto."

"La primera: la función run_import quedó en 71 líneas; las reglas del proyecto en CLAUDE.md piden funciones de menos de 50. La segunda: el router POST no declara explícitamente su status code, y la regla de api punto md exige que todos los endpoints lo hagan. La tercera: admin.js línea 33 tiene 103 caracteres, la regla pide menos de 100."

"Tres cosas que un tester nunca habría encontrado — los tests pasaban verde. Pero el reviewer mirando el código y comparando contra las reglas del proyecto las cazó al toque. Los traté como blockers, hice un commit de cleanup, y recién ahí cerré la branch."
