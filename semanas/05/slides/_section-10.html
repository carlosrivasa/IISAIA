test-driven-development

Rojo, verde, refactor — en ese orden, sin excepciones.

Opener de §10. Cuarta skill del happy-path; pieza s10 iluminada en el roadmap.

"Cuarta skill: test-driven-development. Es la disciplina que opera dentro de cada tarea del plan. Mientras subagent-driven se ocupa de quién ejecuta qué, TDD dice cómo se ejecuta cada implementación: el test va primero, siempre."

El ciclo

1RED ✗ test fails

Escribís un test que describe lo que querés que haga el código. Lo corrés. Lo ves fallar.

→
2GREEN ✓ test passes

Escribís el mínimo código necesario para que el test pase. Nada de extras.

→
3REFACTOR ✓ aún pasa

Con el test en verde, limpiás el código sin romper nada. Después, arrancás otra vuelta con el próximo test.

Una vuelta por cada test. Código sin test previo se borra y se reescribe siguiendo el ciclo.

El ciclo canónico de TDD, con los colores tradicionales. La regla de hierro al final es el delta vs. el TDD "casero": código de prod sin test previo no se "guarda como referencia" — se borra.

"El ciclo tiene tres estaciones. RED: escribís un test que describe lo que querés que haga el código, lo corrés, lo ves fallar. GREEN: escribís el mínimo código necesario para que ese test pase. REFACTOR: con el test en verde, limpiás el código sin romper nada. Y vuelta a empezar con el próximo test."

"La regla de hierro: cualquier código de producción escrito antes de su test se borra y se reescribe siguiendo el ciclo. No se 'adapta', no se 'guarda como referencia' — se borra. Es disciplina, no perfeccionismo."

Cuándo se activa y por qué importa

Cuándo se activa

Por default — sí

Durante la implementación de cualquier feature, bugfix o cambio de comportamiento, antes de escribir código de producción.

Excepciones — con confirmación

Prototipos descartables, código generado y archivos de configuración. Tienen que confirmarse explícitamente; no es el camino default.

Por qué importa

Sin TDD, el agente escribe código que parece correcto y después arma tests que pasan al primer intento. Esos tests post-hoc validan lo que el código hace — no lo que vos querías que hiciera.

En modo subagent-driven, donde el agente corre solo durante una hora sin supervisión, TDD es lo que evita que «ya funciona» termine siendo «compila».

Dos reveals — cuándo + por qué.

Cuándo se activa la skill y por qué la disciplina vale.

"Cuándo se activa: durante la implementación de cualquier feature, bugfix o cambio de comportamiento — antes de escribir código de producción. Si la tarea es agregar un endpoint, la skill arranca con un test del endpoint. Si la tarea es arreglar un bug, arranca con un test que reproduce el bug y falla por la razón correcta."

"Las únicas excepciones, que requieren confirmación explícita, son: prototipos descartables, código generado, y archivos de configuración. Esas tres no van por default por TDD — pero hay que pedirlo a propósito."

"Por qué importa: sin TDD, el agente escribe código que parece correcto y después arma tests que pasan al primer intento. Esos tests post-hoc validan lo que el código hace, no lo que vos querías que hiciera — y por lo tanto pierden su valor como red de seguridad. Vos nunca los viste atrapar un bug. En el contexto de subagent-driven, donde el agente corre solo durante una hora sin supervisión, TDD es exactamente lo que evita que 'ya funciona' termine siendo 'compila'."

Punto crítico

Ver el test fallar antes de implementar.

Si saltás el RED, no sabés si tu test efectivamente está testeando algo: puede tener un typo, puede estar testeando comportamiento que ya existe, puede pasar por la razón equivocada.

La regla central no negociable de TDD: el RED tiene que verse antes de cualquier código de producción.

"Punto crítico: ver el test fallar antes de implementar. Si saltás el RED, no sabés si tu test efectivamente está testeando algo. Puede tener un typo y por eso pasar. Puede estar testeando comportamiento que ya existe en el código. Puede pasar por la razón equivocada — porque la lógica del test está mal, no porque la implementación esté bien."

"Verlo fallar por la razón correcta — feature missing, no typo — es la única garantía de que la lógica del test está bien. Sin esa verificación, el test es una mentira útil que se siente productiva y deja bugs ocultos."

Anti-patrones a evitar

Tests después del código «Implementá X primero, después le agregamos los tests.»
Tests que pasan en RED Aceptar un test verde sin haberlo visto fallar primero — puede no testear nada.
Mocks de todo «Mockeá las dependencias, queremos tests rápidos.»

Tres anti-patrones con la acción concreta que los desencadena.

"Primero: tests después del código. Pasa cuando le decís al agente que implemente algo y después le agreguen los tests. El test resultante valida lo que el código hace, no la intención original, y nunca probaste que atrapa la regresión."

"Segundo: tests que pasan en RED. Esto es lo que pasa cuando aceptás un test verde sin haberlo visto fallar primero. Si el test no falla cuando todavía no implementaste nada, no está testeando lo que decís que testea — y vos no te enteraste."

"Tercero: mocks de todo. Pasa cuando pedís tests con todas las dependencias mockeadas para que corran rápido. Si toda la lógica real está mockeada, el test no falla cuando rompés la lógica real — sólo verifica que los mocks se llamen en el orden esperado."
