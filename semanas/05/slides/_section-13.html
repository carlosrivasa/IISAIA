"Séptima y última skill del happy-path: finishing-a-development-branch. Cuando todas las tareas del plan están implementadas y verificadas, esta skill estructura las opciones de cierre y limpia el workspace según la que elijas. No decide por vos — te muestra el menú que corresponde al contexto y ejecuta la opción elegida con su cleanup correspondiente."

Las cuatro opciones que el skill presenta. La A (PR) es la más común en GitHub Flow.

"La skill te muestra cuatro opciones según el contexto del workspace. La A es abrir un PR — el default en GitHub Flow y la opción más frecuente. Hace push de la branch al remote y crea el PR con título y descripción. La branch queda viva para iterar el feedback de la revisión."

"La B es merge directo a main local. Checkout main, merge fast-forward, corre los tests sobre el resultado mergeado, borra la branch local. Es para trabajo que no necesita revisión humana — refactor interno, fixes triviales, cosas que no van al PR."

"La C es mantener la branch viva. Sin merge, sin push. Es la salida cuando te tenés que ir y el trabajo quedó a mitad de camino — pero con un plan explícito de seguimiento. No es 'la dejo y veo después'."

"La D es descartar todo. Borra la branch y sus commits. Requiere confirmación explícita tipeando 'discard' — para evitar accidentes."

"La regla central: no quedarse en estado intermedio. Una de las cuatro, no 'después decido'. Branches dormidas son ruido en el repo y confusión para quien venga después, incluido vos mismo dentro de dos semanas."

Cuándo se activa y por qué importa

Cuándo se activa

Al final del plan

Después de que verification-before-completion pasó y el trabajo está listo para integrarse.

Cuando te vas a la mitad

Si te tenés que ir y la branch quedó a mitad de camino, la skill te ofrece la opción C — mantenerla viva con un plan claro de seguimiento.

Por qué importa

Sin esta skill el flujo termina en «ya está, mergeá si querés». Esa decisión sin marco produce dos resultados malos típicos: se mergea a main algo que merecía revisión, o la branch queda viva indefinidamente y se convierte en deuda silenciosa.

La skill también ordena el cleanup: mergear primero, después verificar tests sobre el resultado, después remover worktree, y recién al final borrar la branch local. Saltearse ese orden produce errores feos.

Dos reveals — cuándo + por qué.

Cuándo se invoca + qué resultados malos evita.

"Cuándo se activa: al final del plan, después de que verification-before-completion pasó y el trabajo está listo para integrarse. También cuando explícitamente pedís cerrar la sesión y querés decidir qué hacer con lo hecho — por ejemplo si te tenés que ir y la branch quedó a mitad de camino, la skill te ofrece la opción C, mantenerla viva con un plan claro de seguimiento en vez de dejarla flotando sin decisión."

"Por qué importa: sin esta skill el flujo termina en 'ya está, mergeá si querés', y el humano queda decidiendo a ojo qué hacer con una branch que recién parió. Esa decisión sin marco produce dos resultados malos típicos. O se mergea a main algo que merecía revisión, o la branch queda viva indefinidamente y se convierte en deuda silenciosa que después nadie se anima a borrar porque no se acuerda qué tenía."

"La skill también ordena el cleanup: mergear primero, verificar tests sobre el resultado mergeado, después remover el worktree si aplica, y recién al final borrar la branch local. Saltearse ese orden produce errores feos — git branch -d falla si el worktree todavía referencia la branch, y git worktree remove falla en silencio si lo corrés desde adentro del worktree que estás intentando borrar."

El cierre del demo-repo

Primero — verificó

Corrió pytest y git status. Output: 46/46 tests passing, repo normal sin worktree, 13 commits adelante de main.

Después — presentó las 4 opciones
  • Merge a main local
  • Push + abrir PR
  • Dejarla como está
  • Descartar todo el trabajo
Elegí — push solo (sin PR)

Variante de B: git push -u origin feature/import-data, sin abrir PR. La skill ejecutó el push y cerró la sesión con el resumen del trabajo.

Después — pedile a Claude la descripción del PR

Antes de abrir el PR en la UI, podés pedirle a Claude que arme la descripción a partir de los commits de la branch:

«Escribime la descripción del PR para mergear feature/import-data a main. Usá los commits de la branch (git log main..HEAD) y el diff para armar el resumen, los cambios principales y notas de testing.»

Cuatro reveals — uno por step.

Caso real del log. La skill verifica primero, después arma el menú, y ejecuta la elección. El cuarto step (descripción del PR) es el follow-up natural después del push.

"Acá está el cierre real del demo-repo, paso por paso."
