Cuatro comandos cubren el ciclo de vida

/plugin install <plugin>@<marketplace>

Descarga el plugin y registra sus skills, agents y commands. El @<marketplace> desambigua cuando tenés varios registrados.

/plugin update <plugin>

Actualiza a la última versión disponible según el campo version del manifest. Si el manifest no lo declara, cada commit es release nuevo.

/plugin uninstall <plugin>

Desinstala. Por default borra el directorio de datos persistente; pasale --keep-data si vas a reinstalar enseguida.

/plugin list

Lista lo instalado, con versión, marketplace de origen, y si está habilitado o no.

Los 4 comandos del ciclo de vida. Sintaxis exacta + frase corta de qué hace.

"Cuatro comandos cubren todo el ciclo de vida del plugin instalado."

"Install: descarga y registra. La sintaxis es slash plugin install, el nombre del plugin, arroba, el marketplace. Por ejemplo, install formatter at my-marketplace. El arroba marketplace desambigua de qué registro lo están sacando, importante cuando tienen varios registrados."

"Update: actualiza a la última versión disponible según el campo version del manifest que vimos en §2. Si el autor omitió version, Claude Code trata cada commit como release nuevo."

"Uninstall: desinstala. Por default también borra el directorio de datos persistente. Si querés conservarlo — por ejemplo porque vas a reinstalar una versión nueva en seguida — agregás --keep-data."

"List: muestra lo que tenés instalado, con versión, marketplace de origen, y si está habilitado o no. Es el comando que van a usar primero para verificar instalaciones."

El problema que ataca

Sin metodología

Le pedís algo → el agente arranca a tipear código → el desfasaje entre tu intención y su interpretación aparece tarde, con 20 mensajes y archivos commiteados.

Con metodología

Entendé qué construís → escribí el spec → planificá → ejecutá → verificá → mergeá.

Cuatro principios, declarados en el README

Test-Driven Development

Tests primero, siempre. Un test escrito después del código valida lo que el código hace, no lo que vos querías que hiciera.

Sistemático antes que improvisado

Un proceso definido por encima de improvisar. Si no hay un proceso, lo estás inventando cada vez.

Complexity reduction

Simplicidad como objetivo primario. El código más simple que cumple es el correcto.

Evidence over claims

Verificar antes de declarar éxito. «Los tests pasan» sin el output al lado es una afirmación, no un hecho.

Tres approaches arquitecturales

Con las 6 respuestas anteriores, la skill propuso 3 formas de implementar el feature y recomendó la primera.

AEndpoint genéricorecomendada

Un router nuevo backend/routers/imports.py con POST /import/users y POST /import/posts. El parser CSV y el orquestador (validar → dedup → insertar → reportar) viven en backend/services/csv_import.py y son agnósticos del recurso; reciben un adapter por recurso que sabe cómo validar una fila, calcular su dup-key y crearla en el store.

Ventaja: cero duplicación de lógica de errores y rollback. Sumar un tercer recurso después es trivial.

BEndpoint por recurso, lógica inline

POST /users/import dentro de routers/users.py, POST /posts/import dentro de routers/posts.py. Cada uno con su parser, su validación, y su orquestación de rollback.

Trade-off: más simple de leer fila por fila, pero la lógica de rollback y el formato de respuesta se duplica entre los dos routers — y va a divergir con el tiempo.

CEndpoint único con discriminador

Un solo POST /import que toma multipart con resource=users|posts + el file. Un handler para todos los recursos, ramificando por el campo resource.

Trade-off: más fácil de testear como caja negra pero acopla el contrato HTTP — clientes que solo importan users tienen que conocer el discriminador.

Tres conceptos, mínimo viable

1commit

Una foto del repo en un momento. Guarda QUÉ cambió (el diff), QUIÉN lo hizo, CUÁNDO, y POR QUÉ (mensaje del commit).

git add <archivos> git commit -m "tipo: descripción corta"

Regla práctica: commits chicos y frecuentes. Si no se describe en una sola oración corta, probablemente sean dos.

2branch

Una línea paralela de commits. main es la principal. Una branch propia (feature/<descripcion>) no contamina main mientras experimentás.

git checkout -b feature/<descripcion>

Mientras la branch está abierta podés romper, deshacer, reintentar — nadie más se entera.

3pull request

La propuesta de mergear tu branch a main. Tiene título, descripción, y queda visible para el equipo.

No es un comando local — se abre desde la UI de GitHub después del git push.

Hasta que no se mergea, la branch existe pero main no tiene esos cambios.

Tres reveals — uno por concepto.

Refresher de los conceptos base. Suposición: el alumno los vio antes en S04 pero conviene retomarlos antes de las skills que operan sobre git.

"Tres conceptos mínimos. El primero: commit. Una foto del repo en un momento — guarda qué cambió, quién, cuándo, y por qué. Los dos comandos en pantalla: git add para marcar qué archivos van al snapshot, y git commit menos m con un mensaje corto que describe la idea. La práctica es hacer commits chicos y frecuentes — si no se puede describir en una sola oración corta, probablemente sean dos commits."

"Segundo: branch. Una línea paralela de commits. Main es la línea principal del proyecto. Cuando arrancan algo nuevo, abren una branch propia con git checkout menos b seguido del nombre, típicamente con el prefijo feature barra. Dos cosas importantes

Intervención humana

Sólo en dos momentos: cuando ambas etapas del review pasan y se te presenta el resumen, o cuando la skill pide criterio que no está en el plan ni en el spec.

"Cuarta skill: test-driven-development. Es la disciplina que opera dentro de cada tarea del plan. Mientras subagent-driven se ocupa de quién ejecuta qué, TDD dice cómo se ejecuta cada implementación: el test va primero, siempre."

El ciclo

1RED ✗ test fails

Escribís un test que describe lo que querés que haga el código. Lo corrés. Lo ves fallar.

→
2GREEN ✓ test passes

Escribís el mínimo código necesario para que el test pase. Nada de extras.

→
3REFACTOR ✓ aún pasa

Con el test en verde, limpiás el código sin romper nada. Después, arrancás otra vuelta con el próximo test.

Una vuelta por cada test. Código sin test previo se borra y se reescribe siguiendo el ciclo.

El ciclo canónico de TDD, con los colores tradicionales. La regla de hierro al final es el delta vs. el TDD "casero": código de prod sin test previo no se "guarda como referencia" — se borra.

"El ciclo tiene tres estaciones. RED: escribís un test que describe lo que querés que haga el código, lo corrés, lo ves fallar. GREEN: escribís el mínimo código necesario para que ese test pase. REFACTOR: con el test en verde, limpiás el código sin romper nada. Y vuelta a empezar con el próximo test."

"La regla de hierro: cualquier código de producción escrito antes de su test se borra y se reescribe siguiendo el ciclo. No se 'adapta', no se 'guarda como referencia' — se borra. Es disciplina, no perfeccionismo."

Cuándo se activa

Por default — sí

Excepciones — con confirmación

Sin TDD, el agente escribe código que parece correcto y después arma tests que pasan al primer intento. Esos tests escritos después del código validan lo que el código hace — no lo que vos querías que hiciera.

En modo subagent-driven, donde el agente corre solo durante una hora sin supervisión, TDD es lo que evita que «ya funciona» termine siendo «compila».
