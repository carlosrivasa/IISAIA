Sabés escribir cada una de las piezas del runtime de Claude Code.

¿Tenés que escribirlas siempre vos?

Las piezas ya las tenés. Ahora llegan empaquetadas.

Apertura. La pregunta de §1 invierte la dirección de S04: ya no es cómo escribís cada pieza, es cuándo no hace falta que las escribas vos.

"Las siete piezas del runtime ya las saben construir. CLAUDE.md, rules, settings.json, permisos, skills, sub-agents, slash commands — saben qué va en cada una y cómo se ensamblan. Hoy doy vuelta la pregunta: ¿tienen que escribirlas siempre ustedes?"

"Hay gente que ya pasó por los mismos problemas y empaquetó su solución. Anthropic mantiene una colección oficial. Hay autores independientes y empresas que publican sets propios. Alguien ya armó una skill para code review, otra para debug sistemático, otra para mantener un changelog. Buena parte del trabajo que van a hacer no requiere que escriban la skill desde cero — requiere que sepan encontrar la que ya existe."

"Ahora bien, para que alguien pueda compartir su trabajo con vos hace falta un formato común. Sin él aparecen tres complicaciones predecibles. Vamos a verlas."

Cada una de éstas la escribiste vos.

pieza 1 CLAUDE.md
pieza 2 rules
pieza 3 settings.json
pieza 4 permisos
pieza 5 skills
pieza 6 sub-agents
pieza 7 slash commands

Ahora llegan ya armadas.

Recap visual. Las 7 piezas en gris claro para señalar "ya las vieron, no las re-discutimos". Cada card es nominal: nombre + número.

"Estas son las piezas que se construyen una por una. Cada una vive en un archivo o folder distinto del proyecto, cada una tiene su frontmatter, sus convenciones, su lugar de la jerarquía. Saben qué hace cada una y cómo encajan."

"Pero fíjense una cosa: cada vez que arrancan un proyecto nuevo, las tienen que armar de cero — o copiar de un proyecto anterior. Y cuando otro desarrollador tiene una skill genial que ustedes querrían usar, ¿cómo la pasan? ¿Mandan un .zip por mail? ¿Lo suben a un Gist? Esa es la pregunta que abre la sección."

Sin un formato común, aparecen tres complicaciones.

complicación 1

Descubrimiento

No sabés dónde buscar. ¿GitHub? ¿Discord? ¿Un blog? Encontrar algo útil termina dependiendo de que te lo hayan recomendado.

complicación 2

Bajada

Cada autor organiza distinto. Clonás su repo, leés su README, copiás cada folder a la ubicación correcta. Cada vez es un puzzle nuevo.

complicación 3

Mantenimiento

El autor arregla un bug. Vos tenés una foto del día que copiaste. No hay versión, no hay update, no hay forma de enterarte.

Tres reveals — uno por complicación.

Cada complicación se revela una a una para que el alumno la registre por separado antes de ver el conjunto.

"Imaginate que el formato no existe — que cada autor publica su trabajo como mejor le pinta. Aparecen tres complicaciones predecibles."

"El primero es el descubrimiento. Querés saber qué existe pero no sabés dónde buscar. ¿En GitHub? ¿En un blog? ¿En un Discord? Sin un lugar canónico, encontrar una skill útil termina dependiendo de que alguien te la haya recomendado."

"El segundo es la bajada. Encontrás el repo del autor y ahora tenés que clonarlo, leer su README, copiar la carpeta skills/foo/ a tu ~/.claude/skills/, el slash command suelto a otro lado, las rules a otro. Cada autor organiza su trabajo distinto, así que cada vez es un puzzle nuevo."

"El tercero es el mantenimiento. El autor arregla un bug y sube la fix a su repo. Vos no te enterás — tenés una foto que copiaste un día. Cuando algo te falla, no podés ni preguntarte 'qué versión tengo', porque no hay versión: hay un set de archivos que viven en tu disco sin trazabilidad."

La respuesta tiene nombre.

plugin

Un formato estándar que junta esas piezas en un paquete reconocible.

El autor lo publica. Vos lo instalás con un comando.

Cómo es por dentro y cómo llega al disco — viene ya.

Bridge a §2. El nombre del formato + qué resuelve, sin entrar todavía en anatomía ni distribución.

"La respuesta tiene nombre: plugin. Un plugin es el formato estándar que junta esas piezas en un paquete reconocible para Claude Code. El autor lo publica, vos lo encontrás en un marketplace, lo instalás con un comando, lo actualizás con otro. Una vez instalado, sus skills, sus agents y sus slash commands quedan disponibles igual que si los hubieras escrito vos."

"Algunos pueden estar pensando 'y por qué no clono el repo y listo'. Es una pregunta válida y la respondemos en serio en las próximas dos secciones, cuando veamos qué hay adentro de un plugin y cómo se distribuye. Por ahora, quédense con el nombre — plugin — y con la idea de que es lo que resuelve las tres complicaciones. Vamos al primero: qué hay adentro."
