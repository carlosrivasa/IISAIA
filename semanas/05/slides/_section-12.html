verification-before-completion

No digas que funciona si no corriste el comando y mostraste el resultado.

Opener de §12. Sexta skill del happy-path; pieza s12 iluminada en el roadmap.

"Sexta skill: verification-before-completion. La idea es simple. Antes de decir que algo está terminado — un feature, un bugfix, una refactor — corré el comando que lo prueba: pytest, npm test, levantar el server y hacer un curl, lo que sea. Y pegá la salida del comando en el mismo mensaje donde decís 'listo'. Sin esa salida visible, decir 'listo' es opinión, no hecho."

La regla, en una imagen

Claim sin evidencia

los tests pasan

Afirmación. Si no corriste el comando en este turno, es una opinión.

Claim válido

los tests pasan:
$ uv run pytest
======== 46 passed in 1.2s ========

Hecho. Comando corrido en este turno, salida citada al lado.

La diferencia entre creo que funciona y acá está el comando y su output.

Slide central de §12. El contraste visual aterriza la regla.

"La regla es ésta. Sin evidencia, decir 'los tests pasan' es una opinión. Con la salida del comando pegada al lado de la afirmación, es un hecho."

"La diferencia entre opinión y hecho es lo que distingue trabajo terminado de trabajo 'probablemente terminado'. Por eso esta skill fue promovida del resto del cinturón al happy-path: no es para el meticuloso, es lo mínimo para decir 'cerré'."

Cuándo se activa y por qué importa

Cuándo se activa

Antes de cualquier «listo»

Antes del commit «feature listo», antes del PR, antes de cerrar la tarea, antes de pasar a code-review o a finishing-branch.

Si no podés correr el comando

Lo decís explícitamente: «no pude verificar X porque Y». No asumís ni minimizás ni extrapolás desde un check parcial.

Por qué importa

Sin esta skill, el agente claimea éxito porque compiló o porque «los cambios parecen razonables». Esa es la mentira que mata el flujo.

El humano confía, mergea, y descubre el bug en producción o en el siguiente sprint. Compilar es necesario, no suficiente.

Dos reveals — cuándo + por qué.

Cuándo se invoca + qué problema concreto resuelve.

"Cuándo se activa: justo antes de cualquier afirmación de completion. Antes del commit que dice 'feature listo', antes del PR, antes de cerrar la tarea, antes de pasarle el trabajo a code-review o a finishing-branch. Si no podés correr el comando — porque la infra no está, porque depende de algo externo, porque es un cambio que requiere ojo humano — lo decís explícitamente: 'no pude verificar X porque Y'. No asumís, no minimizás, no extrapolás desde un check parcial."

"Por qué importa: sin esta skill, los agentes claimean éxito porque compiló o porque 'los cambios parecen razonables'. Esa es la mentira que mata el flujo. El humano confía, mergea, y descubre el bug en producción o en el siguiente sprint. Compilar es necesario, no suficiente — es la barrera más baja del éxito. Tu trabajo es verificar comportamiento, no sintaxis."

La verificación final del demo-repo

Comando corrido

uv run python -m pytest 2>&1 | tail -3
git rev-parse --show-toplevel
git status --short
git log --oneline main..HEAD | wc -l

Output citado en la afirmación

«46/46 tests passing. Repo normal (sin worktree), 13 commits adelante de main, working tree limpio.»

Caso real: la verificación que el coordinador corrió antes de cerrar la branch. Datos del log.

"Acá está la verificación final del demo-repo, real, del log. Arriba está el comando que el coordinador corrió antes de cerrar: pytest para los tests, git rev-parse para confirmar que estaba en un repo normal sin worktree, git status para el working tree, git log para contar los commits de la branch."

"Abajo está cómo se citó el resultado: '46 de 46 tests passing, repo normal, 13 commits adelante de main, working tree limpio'. No es 'creo que está todo bien' — son cuatro afirmaciones concretas, cada una respaldada por un comando que corrió en el mismo turno. Eso es lo que la skill exige."
