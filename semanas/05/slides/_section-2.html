Definición de la doc oficial

Plugin: directorio autocontenido de componentes que extiende Claude Code.

directorio autocontenido componentes

Las dos palabras importan. Las leemos despacio.

Apertura de §2. La definición literal de la doc; conviene leer las dos palabras clave por separado.

"Un plugin es un directorio autocontenido de componentes que extiende Claude Code con funcionalidad personalizada. Es la definición de la doc oficial, palabra por palabra."

"Vamos a leer las dos partes con cuidado. Primero, 'directorio autocontenido' — todo lo del plugin vive bajo una sola carpeta, sin paths que apunten afuera. Si tenés que clonar tres repos para que el plugin funcione, no es autocontenido."

"Segundo, 'componentes' — no una sola cosa. Adentro del directorio puede haber varias piezas conocidas: skills, agents, hooks, slash commands. Las mismas piezas del runtime que vienen estudiando, ahora juntas bajo un layout que Claude Code ya sabe cargar."

"Vamos a ver cómo se ve por dentro."

La anatomía mínima

mi-plugin/
├── .claude-plugin/
│   └── plugin.json        # manifest opcional con metadata
├── skills/                # una carpeta por skill, con su SKILL.md
├── agents/                # sub-agents en archivos markdown
├── hooks/
│   └── hooks.json         # configuración de hooks
└── commands/              # slash commands como .md sueltos

Las mismas piezas del runtime, ahora bajo un layout estándar.

El árbol mínimo. Cada folder es una de las piezas que ya conocen.

"Adentro del directorio del plugin, el layout que importa al principio es éste. Cada folder lo conocen: skills va al folder skills, agents al folder agents, hooks tiene su archivo hooks.json, commands son los slash commands como markdown sueltos."

"Lo nuevo es el folder con punto adelante: .claude-plugin/. Ahí vive un único archivo, plugin.json, que es el manifest del plugin. Es el archivo opcional que vemos en el próximo slide."

"Si el plugin trae un binario, va en bin/ y queda disponible como comando bare dentro del Bash tool mientras el plugin esté activo. No lo necesitamos hoy, pero está bueno saber que existe."

El manifest mínimo

{
  "name": "mi-plugin",
  "version": "0.1.0",
  "description": "Un plugin de ejemplo",
  "author": { "name": "Equipo", "email": "equipo@ejemplo.com" }
}
De todos los campos, solo name es obligatorio. El resto suma metadata útil. Si no incluís el manifest, Claude Code autodescubre y deriva el nombre desde la carpeta.

Click para resaltar la línea 2 (name).

El manifest. Sólo name es obligatorio. version habilita versionado explícito; sin él, Claude Code cae al SHA del commit.

"El plugin.json es opcional. Si no lo ponés, Claude Code autodescubre componentes en las ubicaciones por defecto y deriva el nombre del plugin desde el nombre de la carpeta. Lo ponés cuando necesitás declarar metadata o apuntar a paths customizados."

"De todos esos campos, el único realmente obligatorio es name. El resto suma metadata útil para que el plugin se encuentre y se versione, pero no es lo que lo hace cargar."

"El campo version es importante por una razón que vemos en la próxima slide: es lo que Claude Code conserva como cache key para saber cuándo ofrecerte una actualización."

Tres reveals — uno por fila.

Cierre conceptual de §1 + §2: el formato resuelve las tres complicaciones, una por una.

"Hay tres cosas que el formato plugin resuelve, y son exactamente las tres complicaciones que aparecieron antes."

"El descubrimiento deja de ser una búsqueda a ciegas porque el formato habilita marketplaces: registros consultables donde el autor publica y vos buscás por nombre. No hace falta seguir blogs ni preguntar en Discord. Hay un lugar canónico al que Claude Code le pregunta '¿qué plugins están disponibles?' — eso lo vemos al detalle en la próxima sección."

"La bajada deja de ser un puzzle nuevo cada vez porque el layout está estandarizado. No importa quién publicó el plugin: las skills siempre van en skills/, los agents en agents/, los hooks en hooks/hooks.json. Vos no decidís dónde se instala cada cosa porque ya está decidido."

"El mantenimiento deja de ser una foto inmóvil porque el plugin lleva versión explícita. Si el plugin.json declara version 2.1.0, ese string es la versión que Claude Code conserva como cache key. Cuando el autor sube 2.1.1, vos sabés exactamente qué cambió. Si el autor prefiere iterar más rápido y omite el campo, Claude Code cae al commit SHA del repo como versión, y cada commit es tratado como release nuevo. Dos modos, los dos explícitos, los dos visibles para el que instala."

No tenés que adivinar qué existe — hay marketplaces.

No tenés que aprender el layout de cada autor — el layout es uno solo.

No tenés que rastrear cambios a mano — hay versión.

Lo único que falta es el comando que instala. Viene ya.
