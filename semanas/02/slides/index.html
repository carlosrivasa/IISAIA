Dos formas de escribir la misma página

HTML semántico

Cada bloque usa la etiqueta que describe qué es: <header>, <nav>, <main>, <article>, <section>, <footer>.

Cualquiera de ellas puede llevar también un atributo class — la etiqueta da el rol estructural; la clase agrega el gancho para CSS o JavaScript.


<nav>
  <ul><li>Inicio</li></ul>
</nav>
<main>
  <h1>Hola mundo</h1>
</main>

Sopa de divs

<div> es el contenedor genérico de HTML: un bloque sin significado propio. Existe para agrupar contenido cuando ninguna etiqueta semántica encaja.

La sopa de divs es cuando se lo usa para todo. El rol vive en el atributo class — un nombre que vos inventás, que solo CSS y JavaScript leen. El navegador y los buscadores no lo entienden como rol estructural.


<div class="top-bar">
  <div class="link-item">Inicio</div>
</div>
<div class="content-body">
  <div class="headline">Hola mundo</div>
</div>

Las dos versiones se ven exactamente igual cuando el navegador las dibuja. Pero el código es muy distinto.

"La primera se llama HTML semántico. Las etiquetas hablan: alguien que lee este código, sin necesidad de ejecutarlo, ya sabe que arriba hay una barra de navegación con un enlace a Inicio, y abajo el contenido principal con el título 'Hola mundo'."

"Una aclaración importante antes de seguir: las etiquetas semánticas no excluyen al atributo class. Cualquiera de ellas puede llevar una clase — vas a ver esto en la próxima sección, cuando agreguemos CSS. Por ejemplo, un nav con clase 'principal' es perfectamente válido y común. La etiqueta dice qué es; la clase es un nombre interno que vos elegís para que CSS o JavaScript puedan engancharse a ese elemento. No son alternativas, son complementos. La sopa de divs no es 'usar class' — es usar class como única descripción del rol, sin etiqueta semántica que la respalde."

"La segunda versión está hecha enteramente con divs. Pausemos un segundo en qué es un div: es el contenedor genérico de HTML, un bloque sin significado propio. Existe a propósito en el lenguaje, para agrupar contenido cuando ninguna etiqueta semántica encaja — es una herramienta válida. Pero la 'sopa de divs' es lo que pasa cuando se la usa para todo. Mirá los nombres: top-bar, link-item, content-body, headline. Esos no son etiquetas, son clases. El atributo class es un nombre que vos inventás y que solo el CSS y el JavaScript van a leer para encontrar el elemento. El navegador no lo entiende como un rol estructural; los buscadores tampoco; los lectores de pantalla tampoco. Para todos ellos, es solamente un div anónimo."

La próxima slide explica por qué importa."

Por qué importa la diferencia

Dimensión HTML semántico Sopa de divs
Claridad Las etiquetas describen el contenido. Leés el código y entendés la página. Todos los bloques se ven iguales. El propósito hay que adivinarlo desde las clases.
Accesibilidad Los lectores de pantalla saltan directo a <nav> o <main>: existen como puntos de referencia nativos. No hay puntos de referencia. Para suplirlo hay que agregar atributos role y aria-* a mano.
SEO (Search Engine Optimization) Buscadores entienden la jerarquía y la importancia del contenido. El buscador ve cajas sin estructura clara: indexación más pobre.
Mantenimiento Leer <article> y saber qué contiene. Leer divs anidados con clases inventadas y reconstruir la intención del autor.

La sopa de divs no aparece por mala fe: aparece por hábito viejo, por frameworks que la fomentan, o porque tirar un <div> es más rápido que pensar qué etiqueta corresponde.

Cinco clicks: una fila por click (claridad → accesibilidad → SEO → mantenimiento), después la nota de cierre.

Tradeoffs entre HTML semántico y sopa de divs en cuatro dimensiones. La nota inferior justifica por qué la sopa de divs sigue siendo común.

"La pregunta natural acá es: si las dos páginas se ven iguales y las dos funcionan, ¿qué importa cuál usemos? Importa por cuatro dimensiones, una por una."

"Claridad. El código semántico se lee. Si abren un archivo HTML y lo primero que ven son sus etiquetas, entienden la página antes de ejecutarla. Con sopa de divs, todos los bloques se ven iguales y tienen que ir adivinando qué hace cada uno desde el nombre de la clase."

"Accesibilidad. Los lectores de pantalla, que son la herramienta que usan personas con discapacidad visual para navegar la web, usan las etiquetas semánticas como puntos de referencia: pueden saltar directo a la navegación, directo al contenido principal. Con sopa de divs no hay puntos de referencia. Para imitarlos hay que agregar atributos extra a cada div — role='navigation', aria-label='...' y similares, los que define la especificación ARIA. Es trabajo manual, casi nadie lo hace bien, y un ARIA mal escrito miente al lector de pantalla, lo cual es peor que no tenerlo."

"SEO. Los buscadores como Google leen el HTML para entender de qué trata la página. Si tenés etiquetas semánticas, el buscador entiende la jerarquía: el título principal, las secciones, la navegación. Con sopa de divs, ve cajas sin estructura, y la indexación es más pobre. Para un sitio que necesita ser encontrado por Google, esto se traduce en tráfico."

"Mantenimiento. Dentro de seis meses alguien va a tener que tocar este código — vos mismo, otro desarrollador, o una IA. Leer un article y saber qué contiene es trivial. Leer divs anidados con clases inventadas y reconstruir la intención original del autor es una pérdida de tiempo."

"La pregunta final: si todo esto es tan obvio, ¿por qué la sopa de divs sigue siendo tan común? Por tres razones, ninguna buena pero todas reales. Hábito viejo: las versiones anteriores de HTML no tenían estas etiquetas semánticas, así que el div era la herramienta universal. Frameworks: muchas librerías populares de CSS o de componentes generan divs anidados por defecto. Y planificación: tirar un div es más rápido que pensar dos segundos qué etiqueta corresponde."

Si las usás bien, el resto se ordena solo. No memorices: reconocé.

Los siete contenedores estructurales. La tabla es de referencia, no se lee entera.

"Existen como cien etiquetas en HTML. Pero con estas siete arma cualquier página razonable. Header arriba, main al medio, sections temáticas adentro, articles para piezas autocontenidas, nav para enlaces, footer abajo, aside para lo tangencial. No las memorices: reconocelas. Las dos familias de elementos que viven adentro de los contenedores: contenido e interacción.

"Ya vimos los siete contenedores estructurales. Adentro de ellos van dos familias de elementos: lo que la página muestra y lo que la página le pide al usuario."

"Empecemos por lo que se muestra. Encabezados con jerarquía — h1 a h6, donde h1 es el título principal de la página. Párrafos con la etiqueta p. Listas: ul para no ordenadas, ol para ordenadas, y li para cada ítem. Enlaces con la etiqueta a, que apunta a otra página o a un ancla interna. E imágenes con img — y siempre con el atributo alt, que es el texto que describe la imagen para alguien que no la puede ver."

"Y cuando la página le pide algo al usuario, aparecen las cuatro etiquetas de interacción: button para botones clickeables, input para campos de entrada — el tipo se controla con el atributo type, sea texto, número, email o checkbox. Form agrupa inputs y los envía juntos. Label es la etiqueta visible asociada a cada input. Repito lo de la nota al pie: sin label, el input no es accesible para un lector de pantalla; sin alt, la imagen no es accesible para nadie. Estos son los nombres con los que vas a pedir piezas concretas en cualquier prompt."

Cómo le pedís estructura a una IA

Prompt sin vocabulario

"Necesito una página de contacto. Tiene un menú arriba, un formulario para nombre, email y mensaje, y un pie con mi copyright."

La intención está clara, pero la estructura la decide la IA. A veces acierta, a veces no — y nunca sabés cuál vas a recibir.

Prompt con vocabulario

"Necesito una página de contacto. Estructura: <header> con <nav>, <main> con un <form> para nombre, email y mensaje, y <footer> con copyright."

Misma intención, cinco etiquetas nombradas. La IA no improvisa; vos podés iterar y auditar.

Vago = lotería.   Específico = determinismo.

Cierre de §2. El vocabulario sigue siendo necesario aunque los LLMs modernos suelan acertar en casos textbook.

"Acá es donde el vocabulario te paga. Mirá los dos prompts: la intención es exactamente la misma. Una página de contacto con menú, formulario y pie."

"Quiero ser honesto con un detalle. Si vos hoy le pasás el prompt de la izquierda a Claude, a ChatGPT o a Gemini, lo más probable es que te devuelva HTML semántico, con header, nav, main y footer correctos. Las páginas de contacto son un caso textbook que los modelos vieron diez mil veces. Hace cuatro años un prompt así te tiraba sopa de divs; hoy ya no, en la mayoría de los casos. Entonces la pregunta legítima es: si la IA acierta sola, ¿para qué necesito el vocabulario?"

"Por tres razones que sobreviven a esa mejora. Primero, determinismo. La IA acertó esta vez con la página de contacto. Probá pedirle un dashboard, una tarjeta de perfil con estadísticas, un hero con tres columnas — algo menos común — y la respuesta empieza a variar entre semántico, sopa de divs, y mezclas raras. Con vocabulario, no varía."

"Segundo, iteración. Tu próximo prompt va a ser 'movele la nav a la derecha' o 'convertí el formulario en un grid de dos columnas'. Si no nombraste nav y form en el primer prompt, la IA tiene que adivinar a qué te referís y vos tenés que escribir más. El vocabulario hace barato cada paso siguiente de la conversación."

"Tercero, auditoría. El vocabulario no es solo para que la IA te entienda; es para que vos puedas leer y verificar lo que devolvió. Si te llega un header con cinco divs anidados de más, vas a notarlo solamente si tenés los nombres en la cabeza."

Primitiva 1 — El modelo de caja

margin
border
padding

hola

Todo elemento HTML es una caja con cuatro capas concéntricas.

padding: espacio adentro · border: el borde · margin: espacio entre cajas 

<button class="apretado">Aceptar</button>
<button class="estandar">Aceptar</button>
<button class="generoso">Aceptar</button>

Resultado — pares de botones idénticos. Mirá el tamaño (padding), el grosor del borde (border) y el espacio entre hermanos (margin):

apretado 
.apretado { padding: 2px 6px; margin: 0; border: 1px solid; border-radius: 2px; }
estándar 
.estandar { padding: 8px 14px; margin: 8px 0; border: 1px solid; border-radius: 4px; }
generoso 
.generoso { padding: 14px 24px; margin: 20px 0; border: 3px solid; border-radius: 8px; }

padding y margin son las palabras de "respirar".

Explicar el diagrama y el HTML primero. Después un click revela el ejemplo de pares.

Box model. La caja como unidad universal del CSS. El ejemplo aparece con un click para no abrumar mientras se introduce el concepto.

"Todo elemento HTML es una caja. No 'casi todo': todo. Cada caja tiene cuatro capas concéntricas, como una cebolla. Adentro de todo está el contenido — el texto, la imagen, lo que sea. Alrededor del contenido, padding: el espacio interno entre el contenido y el borde. Después border, el borde mismo. Y afuera de todo, margin: el espacio externo que separa esta caja de las que la rodean."

"Y arriba en el código tenemos el patrón HTML: tres botones, cada uno con una clase distinta — apretado, estándar y generoso. Cada clase va a tener una regla CSS con valores distintos para esas cuatro capas. Vamos a verlas en concreto."

"Para que se vea bien qué hace cada capa, en cada fila puse dos botones idénticos uno arriba del otro. Y en cada fila los tres valores del box model — padding, border y margin — son distintos, así que pueden ver las tres capas trabajando al mismo tiempo."

"Apretado: padding chico de 2 por 6, border de 1 píxel, esquinas casi rectas. El botón sale chico, casi sin aire interno. Y como el margin es cero, los dos hermanos están pegados — sin ningún espacio entre ellos."

"Estándar: padding razonable de 8 por 14, border de 1 píxel, esquinas suaves. El botón es más grande, cómodo de leer y clickear. Margin de 8 píxeles, así que entre los dos hermanos aparece un espacio claro."

"Generoso: padding grande de 14 por 24, border de 3 píxeles, esquinas más redondeadas. El botón pesa visualmente, atrae la atención. Margin de 20 píxeles, y la separación entre hermanos es notoriamente más grande que en estándar."

"Una aclaración por si lo pregunta alguien: en las reglas también ven border-radius con valores distintos en cada variante. No es parte del box model — el box model son las cuatro capas que vimos. Border-radius es una propiedad aparte que redondea las esquinas del border. La metí en el ejemplo para que la progresión apretado-estándar-generoso se vea más natural visualmente, pero conceptualmente no es una quinta capa."

"Tres palabras que necesitás para pedirle a la IA cualquier ajuste visual: padding adentro, border alrededor, margin entre cajas. Con esas tres, no hay ambigüedad."

"La diferencia entre una página apretada y una que respira se decide en padding y margin. Sin esos dos nombres, vas a decirle a la IA 'dale más aire' y la IA va a tirar al azar — a veces padding, a veces margin, a veces line-height. Con esos dos nombres, el cambio es exacto."

Primitiva 4 — Variables CSS

La primitiva del LLM.

<button class="btn">Aceptar</button>
<a class="link">Saber más</a>
<span class="badge">Nuevo</span>
:root { --color-accent: ???; }
.btn   { background: var(--color-accent); color: #fff; }
.link  { color: var(--color-accent); }
.badge { border: 1px solid var(--color-accent); color: var(--color-accent); }

Una sola variable, tres elementos. Cambiar el valor de --color-accent reviste todo:

--color-accent: #10b981 Saber más Nuevo
--color-accent: #f59e0b Saber más Nuevo

Sin variables: cambiar el acento son N reemplazos de hex con riesgo de romper algo. Con variables: una línea.

"Arriba tenemos el patrón. Tres elementos en HTML: un button, un enlace, y un badge — un span pequeño que diría algo como 'Nuevo'. En el CSS, una variable --color-accent definida en :root, y tres reglas que la usan: el botón la usa de fondo, el enlace de color de texto, el badge de color de borde y de texto. El valor de la variable está vacío por ahora — es justamente lo que vamos a variar."

"Click. Acá tienen tres filas, una por cada valor de --color-accent. La fila de arriba: indigo. Botón con fondo indigo, link en indigo, badge con borde indigo. La del medio: esmeralda. Botón verde, link verde, badge verde. La de abajo: ámbar. Todo en ámbar. Lo único que cambia entre filas es la primera línea del CSS: el valor de --color-accent. Tres elementos, una sola variable, todo enganchado."

"Ese es el patrón que hace que las modificaciones futuras sean baratas. Sin variables, cambiar el acento son N reemplazos del hex en cada regla, con riesgo de romper algo si el mismo hex se usaba para otra cosa. Con variables, una línea cambia todo. Por eso la llamo 'la primitiva del LLM' — es lo que más le facilita el trabajo a la IA cuando tiene que iterar tu código."

El triángulo dinámico

Estado

El dato que la página recuerda. Vive en variables, no en pantalla.

Ej.: count = 0, items de una todo list, modo dark/light.

DOM

El HTML, pero vivo. Un árbol en memoria que JavaScript puede mutar.

Mutar el DOM cambia lo que el usuario ve sin recargar.

Eventos

La voz del usuario. Click, tecla, scroll, input.

Un listener escucha; un handler responde.
