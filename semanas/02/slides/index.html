Apertura de §2. La capa de estructura activa en el roadmap.

"Arrancamos con la primera capa: estructura. El HTML no se trata de hacer que algo se vea bien — eso es trabajo del CSS, lo vamos a ver después. El HTML se trata de nombrar las partes de tu página con la etiqueta correcta. Si la etiqueta es precisa, todo lo demás se vuelve más fácil."

¿Estas dos líneas son iguales?

<header>Mi sitio</header>
<div class="header">Mi sitio</div>

Visualmente: idénticas.

Para el navegador, los lectores de pantalla, los buscadores y la IA: no lo son.

La primera dice "esto es un encabezado". La segunda dice "esto es una caja, le puse de apodo header, podría ser cualquier cosa". El outline del documento solo se construye con etiquetas semánticas.

Hook de §2. Construye la tensión: visualmente iguales, semánticamente distintas.

"Compará estas dos líneas. La de la izquierda usa la etiqueta header. La de la derecha es un div con la palabra header como nombre. La pregunta: ¿son lo mismo?"

"Visualmente, sí. Las dos te van a renderizar el mismo texto en la misma posición."

"Pero para el navegador, los lectores de pantalla, los buscadores y la IA que después tenga que tocar tu código, no son la misma cosa."

"La primera dice 'esto es un encabezado'. La segunda dice 'esto es una caja anónima'. El outline del documento, que es el árbol de qué contiene qué, solo se construye con la primera. Con la segunda, no existe."

Siete etiquetas arman el esqueleto

Etiqueta Qué representa

<main>contenido principal de la página, único por documento
<section>bloque temático con título propio

Si las usás bien, el resto se ordena solo. No memorices: reconocé.

Los siete contenedores estructurales. La tabla es de referencia, no se lee entera.

"Existen como cien etiquetas en HTML. Pero con estas siete arma cualquier página razonable. Header arriba, main al medio, sections temáticas adentro, articles para piezas autocontenidas, nav para enlaces, footer abajo, aside para lo tangencial. No las memorices: reconocelas. Cuando la IA te devuelva HTML, querés ver estos nombres apareciendo. Si solamente ves divs, hay un problema."

Las dos familias de elementos que viven adentro: contenido e interacción.

"Adentro de los contenedores va el contenido en sí. Encabezados con jerarquía, párrafos, listas, links, imágenes. Y cuando la página le pide algo al usuario, aparecen las cuatro de interacción: button, input, form, label. La nota al pie es importante: sin label el input no es accesible para un lector de pantalla, y sin alt la imagen no es accesible para nadie. Estos son los nombres con los que vas a pedir piezas concretas."

CSS es enorme. Para dirigir a la IA alcanza con cuatro perillas.

Apertura de §3. Capa de estilo activa.

"Si el HTML nombra las partes, el CSS responde a la pregunta cómo se ven. CSS es enorme: cientos de propiedades, módulos enteros que llenan libros. Pero para dirigir a una IA no necesitás ese inventario. Necesitás cuatro perillas. Con eso alcanzás para describir el 90% de las decisiones visuales del curso."

Perilla 1 — El modelo de caja

margin
border
padding
content
hola

Todo elemento HTML es una caja con cuatro capas concéntricas.

La diferencia entre una página "apretada" y una que "respira" se decide en padding y margin. Si no tenés esos dos nombres, no podés pedir el cambio.

button {
  padding: 12px 20px;
  margin: 8px 0;
}

Box model. Las dos palabras de "respirar": padding y margin.

"Todo elemento HTML es una caja. No 'casi todo': todo. Y cada caja tiene cuatro capas concéntricas como una cebolla. Adentro está el contenido. Después padding, que es el espacio interno entre el contenido y el borde. Después border, el borde mismo. Y afuera margin, el espacio externo que separa esta caja de las que la rodean. La diferencia entre una página apretada y una que respira se decide en padding y margin. Sin esos dos nombres, vas a decirle a la IA 'dale más aire' y la IA va a tirar al azar."

Perilla 2 — Layout: una o dos dimensiones

Flexbox — 1D

Acomoda en una dirección: fila o columna.

.toolbar {
  display: flex;
  justify-content: space-between;
  align-items: center;
  gap: 16px;
}

Caso típico: nav-bar, toolbar, grupo de botones.

Grid — 2D

Tablero: filas y columnas a la vez.

.gallery {
  display: grid;
  grid-template-columns: 1fr 1fr 1fr;
  gap: 24px;
}

Caso típico: galería, dashboard, cuadrícula de productos.

Nombrar flex o grid en el prompt evita que la IA improvise con inline-block o floats antiguos.

Las dos respuestas dominantes a "cómo se acomodan las cajas".

"La pregunta principal del CSS no es de qué color son las cosas. Es cómo se acomodan las cajas entre sí. Y en 2026 hay dos respuestas que tenés que saber nombrar. Flex sirve cuando querés acomodar elementos en una sola dirección. Grid sirve cuando el layout es un tablero. Si nombrás flex o grid en el prompt, el código sale bien al primer intento. Si decís 'quiero que estén una al lado de la otra', la IA improvisa con cosas viejas y vas a iterar tres veces."

Si querés cambiarlo, la IA tiene que buscar veinte ocurrencias y reemplazarlas a mano. A la derecha, una sola variable. Cambiar la paleta es cambiar una línea. Esa es la perilla que querés que tu código tenga siempre."

Cuando una página hace algo, hay un triángulo de tres conceptos en juego al mismo tiempo.

Apertura de §4. Capas 3 y 4 activas — comportamiento y estado son la misma sección porque van juntos.

"Una página estática se entiende sola: hay HTML, lo dibuja el navegador, listo. Pero en cuanto pasa cualquier otra cosa cuando el usuario interactúa, ya sea un click que suma uno, un input que filtra una lista, o un botón que cambia el tema, entran tres piezas en juego al mismo tiempo. Comportamiento y estado son la misma sección porque van pegados. Las tres tienen nombre, y saberlos es la diferencia entre pedirle a una IA hacé que funcione y pedirle algo que el modelo pueda traducir directo a código."

El triángulo dinámico

Estado

El dato que la página recuerda. Vive en variables, no en pantalla.

Ej.: count = 0, items de una todo list, modo dark/light.

DOM

El HTML, pero vivo. Un árbol en memoria que JavaScript puede mutar.

Mutar el DOM cambia lo que el usuario ve sin recargar.

Eventos

La voz del usuario. Click, tecla, scroll, input.

Un listener escucha; un handler responde.

Revelar los tres conceptos en orden, uno por click.

Las tres entidades del triángulo dinámico, una por columna.

"Tres conceptos."

"El estado es el dato que la página recuerda: vive en variables, no en pantalla. Es invisible hasta que alguien decide dibujarlo."

"El DOM es el HTML pero vivo: un árbol en memoria que JavaScript puede modificar. Cambiarlo cambia lo que el usuario ve sin recargar."

"Los eventos son la voz del usuario: cualquier cosa que el usuario hace genera uno. Para escuchar uno registrás un listener; la función que se ejecuta es el handler. Las tres palabras son las que la IA necesita para no inventar atajos malos."

Cierre de §4. La palabra "estado" en el prompt cambia el código que sale.

"Sin nombrar el estado, la IA tiene tres atajos disponibles, y los tres son malos. Puede guardar el contador en un atributo del DOM. Puede usar una variable global flotante. Puede mezclar el dato con su representación en una sola variable. Si en cambio nombrás estado en el prompt, le estás diciendo: esto es un dato, separalo del DOM, ponelo donde corresponde. La palabra estado hace ese trabajo sola. Próxima capa, la última de la anatomía: cómo metemos todo esto en un solo archivo."
