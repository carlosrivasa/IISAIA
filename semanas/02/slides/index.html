Dos formas de escribir la misma página

HTML semántico

Cada bloque usa la etiqueta que describe qué es: <header>, <nav>, <main>, <article>, <section>, <footer>.

Cualquiera de ellas puede llevar también un atributo class — la etiqueta da el rol estructural; la clase agrega el gancho para CSS o JavaScript.


<nav>
  <ul><li>Inicio</li></ul>
</nav>
<main>
  <h1>Hola mundo</h1>
</main>

Sopa de divs

<div> es el contenedor genérico de HTML: un bloque sin significado propio. Existe para agrupar contenido cuando ninguna etiqueta semántica encaja.

La sopa de divs es cuando se lo usa para todo. El rol vive en el atributo class — un nombre que vos inventás, que solo CSS y JavaScript leen. El navegador y los buscadores no lo entienden como rol estructural.


<div class="top-bar">
  <div class="link-item">Inicio</div>
</div>
<div class="content-body">
  <div class="headline">Hola mundo</div>
</div>

Las dos versiones se ven exactamente igual cuando el navegador las dibuja. Pero el código es muy distinto.

Por qué importa la diferencia

Dimensión HTML semántico Sopa de divs
Claridad Las etiquetas describen el contenido. Leés el código y entendés la página. Todos los bloques se ven iguales. El propósito hay que adivinarlo desde las clases.
Accesibilidad Los lectores de pantalla saltan directo a <nav> o <main>: existen como puntos de referencia nativos. No hay puntos de referencia. Para suplirlo hay que agregar atributos role y aria-* a mano.
SEO (Search Engine Optimization) Buscadores entienden la jerarquía y la importancia del contenido. El buscador ve cajas sin estructura clara: indexación más pobre.
Mantenimiento Leer <article> y saber qué contiene. Leer divs anidados con clases inventadas y reconstruir la intención del autor.

La sopa de divs no aparece por mala fe: aparece por hábito viejo, por frameworks que la fomentan, o porque tirar un <div> es más rápido que pensar qué etiqueta corresponde.

"La pregunta natural acá es: si las dos páginas se ven iguales y las dos funcionan, ¿qué importa cuál usemos? Importa por cuatro dimensiones, una por una."

"Claridad. El código semántico se lee. Si abren un archivo HTML y lo primero que ven son sus etiquetas, entienden la página antes de ejecutarla. Con sopa de divs, todos los bloques se ven iguales y tienen que ir adivinando qué hace cada uno desde el nombre de la clase."

"Accesibilidad. Los lectores de pantalla, que son la herramienta que usan personas con discapacidad visual para navegar la web, usan las etiquetas semánticas como puntos de referencia: pueden saltar directo a la navegación, directo al contenido principal. Con sopa de divs no hay puntos de referencia. Para imitarlos hay que agregar atributos extra a cada div — role='navigation', aria-label='...' y similares, los que define la especificación ARIA. Es trabajo manual, casi nadie lo hace bien, y un ARIA mal escrito miente al lector de pantalla, lo cual es peor que no tenerlo."

"SEO. Los buscadores como Google leen el HTML para entender de qué trata la página. Si tenés etiquetas semánticas, el buscador entiende la jerarquía: el título principal, las secciones, la navegación. Con sopa de divs, ve cajas sin estructura, y la indexación es más pobre. Para un sitio que necesita ser encontrado por Google, esto se traduce en tráfico."

"Mantenimiento. Dentro de seis meses alguien va a tener que tocar este código — vos mismo, otro desarrollador, o una IA. Leer un article y saber qué contiene es trivial. Leer divs anidados con clases inventadas y reconstruir la intención original del autor es una pérdida de tiempo."

"La pregunta final: si todo esto es tan obvio, ¿por qué la sopa de divs sigue siendo tan común? Por tres razones, ninguna buena pero todas reales. Hábito viejo: las versiones anteriores de HTML no tenían estas etiquetas semánticas, así que el div era la herramienta universal. Frameworks: muchas librerías populares de CSS o de componentes generan divs anidados por defecto. Y planificación: tirar un div es más rápido que pensar dos segundos qué etiqueta corresponde."

Sintaxis CSS

Una regla CSS es la unidad básica de una hoja de estilos: una instrucción que le dice al navegador "encontrá este elemento y aplicale estos estilos".

  color: white;
  padding: 8px;
}

"Antes de hablar de las cuatro primitivas, démosle nombre a las partes del código CSS que vamos a ver durante todo el curso."

"Lo primero: ¿qué es una regla CSS? Es la unidad básica de una hoja de estilos. Una sola instrucción que le dice al navegador 'encontrá este elemento del HTML y aplicale estos estilos'. Cualquier hoja de estilos, por más grande que sea, es una colección de reglas como esta. Mirá la que tenemos en pantalla."

"En azul-verdoso tenés el selector — punto-btn. El selector es lo que apunta a un elemento del HTML; en este caso, a cualquier elemento que tenga la clase btn. Si fuera un selector de etiqueta, sería simplemente button, sin punto."

"En verde tenés las propiedades — color y padding. Una propiedad es qué aspecto del elemento querés cambiar. Hay cientos en CSS, pero las que importan para este curso son las de las cuatro primitivas que vienen después."

"En ámbar tenés los valores — white y 8px. El valor es qué valor concreto le asignás a la propiedad. Cada propiedad acepta un tipo de valor distinto: color acepta nombres o hex, padding acepta píxeles, etcétera."

"Y dos términos más: una declaración es la combinación propiedad-dos-puntos-valor-punto-y-coma — color dos puntos white punto y coma es una declaración. Una regla es el selector más todas las declaraciones encerradas entre llaves. Selector, propiedad, valor, declaración, regla. Cinco palabras y ya entendés la sintaxis básica del CSS."

Box model. La caja como unidad universal del CSS. El ejemplo aparece con un click para no abrumar mientras se introduce el concepto.

"Todo elemento HTML es una caja. No 'casi todo': todo. Cada caja tiene cuatro capas concéntricas, como una cebolla. Adentro de todo está el contenido — el texto, la imagen, lo que sea. Alrededor del contenido, padding: el espacio interno entre el contenido y el borde. Después border, el borde mismo. Y afuera de todo, margin: el espacio externo que separa esta caja de las que la rodean."

"Y arriba en el código tenemos el patrón HTML: tres botones, cada uno con una clase distinta — apretado, estándar y generoso. Cada clase va a tener una regla CSS con valores distintos para esas cuatro capas. Vamos a verlas en concreto."

La idea es simple: en lugar de escribir el mismo color o el mismo número de píxeles en veinte lugares distintos, lo definís una sola vez con un nombre, y después lo usás por nombre en cada lugar que lo necesites."

"Ahí está la razón por la que llamo a las variables 'la primitiva del LLM'. El dato detrás del HTML. Vive en una variable, no en pantalla.

El árbol de nodos del HTML en memoria. JavaScript apunta al span y lo muta.

Cómo se ve un single-file

<!DOCTYPE html>                       <!-- modo HTML5 estricto -->
<html lang="es">                       <!-- raíz del documento; lang ayuda a accesibilidad y SEO -->
<head>                                 <!-- metadatos del documento, no se renderiza -->
  <meta charset="UTF-8">               <!-- soporta tildes, eñes y otros caracteres -->
  <title>Contador</title>              <!-- texto de la pestaña del navegador -->

  <!-- Capa 2: Estilo (acá porque el CSS debe cargar antes del paint) -->
  <style>
    #count  { font-size: 2rem; font-family: monospace; }    /* estilos del span del número */
    button  { background: #f59e0b; color: #0f172a;          /* color del botón */
              padding: 8px 14px; border: none; border-radius: 4px; }
  </style>
</head>
<body>                                 <!-- contenido visible de la página -->
  <!-- Capa 1: Estructura -->
  <span id="count">0</span>            <!-- el JS va a escribir acá el valor de count -->
  <button id="inc">+1</button>         <!-- el JS va a escuchar el click acá -->

  <!-- Capas 3 y 4: Comportamiento + Estado (al final del body para que el DOM ya exista) -->
  <script>
    let count = 0;                                          // estado: la verdad del contador
    const span = document.querySelector('#count');          // referencia al span del DOM
    const btn  = document.querySelector('#inc');            // referencia al botón del DOM
    btn.addEventListener('click', () => {                   // listener: corre cada vez que hacen click
      count++;                                              // mutar estado
      span.textContent = count;                             // mutar DOM (refleja el estado)
    });
  </script>
</body>
</html>
