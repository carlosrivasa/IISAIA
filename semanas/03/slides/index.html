Anatomía de un par request / response

Cliente arma un request. Servidor arma una response. Entre los dos cabe casi todo lo que hace internet hoy. 

GET /projects/4 HTTP/1.1
Host: api.example.com

— — — respuesta — — —

HTTP/1.1 200 OK
Content-Type: application/json

{ "id": 4, "name": "Tesis" }

Request — cuatro partes

método · URL · headers · body opcional

Response — tres partes

status code · headers · body

Anatomía concreta. Se lee el bloque línea por línea mientras está en pantalla, sin fragments.

"Ese idioma común se llama HTTP, y es un protocolo de pedido y respuesta. El cliente arma un request, el servidor arma una response, y entre los dos cabe casi todo lo que hace internet hoy. Mirá un par concreto en pantalla."

"Arriba, el request, el pedido. Tiene cuatro partes. Primero el método: GET. Es el method, ya viene definido por el protocolo, no es una palabra que vos inventás. Después la URL: barra projects barra cuatro. Es a qué recurso le estás hablando — querés el proyecto número cuatro. Después HTTP barra uno punto uno, que es la versión del protocolo. Y abajo los headers: metadatos. Acá tenemos uno solo, Host, que dice a qué dominio le estoy pidiendo: api.example.com. Y este request en particular no lleva body — los GET típicamente no llevan, porque solo están pidiendo información, no mandando contenido."

"Abajo, la response. Tres partes. Primero el status code: doscientos OK. Eso quiere decir 'todo bien, acá tenés lo que pediste'. Después los headers: Content-Type application/json, que dice 'lo que te mando es JSON'. Y abajo el body: el contenido en sí, en este caso un objeto con id cuatro y nombre Tesis."

"Ese ida y vuelta, repetido millones de veces por segundo, es internet. Y el vocabulario para describirlo es exactamente lo que tenés en pantalla. Cuando le dictés un endpoint a la IA, vas a tener que nombrar estas piezas — el método, el path, qué headers esperás, qué viene en el body. Si no las nombrás, el modelo elige por vos."

POST

PATCH

modificar parte de un recurso

borrar un recurso

La palabra method engaña: no son acciones libres que componés vos. Son los pocos sustantivos del trabajo — operaciones que se repiten en toda API.

Ese es todo el repertorio que vas a necesitar el 95% del tiempo.

Cinco methods como repertorio cerrado. La aclaración terminológica importa: no son acciones libres.

"HTTP define un puñado de métodos, llamados methods. La palabra engaña un poco: no son acciones que vos componés libremente. Son los pocos sustantivos del trabajo que pedís — un repertorio cerrado de operaciones que aparecen en toda API."

"Cinco que importan. GET: leer un recurso, sin efectos. Pedís información y nada cambia del lado del servidor. POST: crear un recurso nuevo. Mandás contenido y aparece algo que antes no existía. PUT: reemplazar un recurso entero por la versión que mandás — la versión vieja desaparece. PATCH: modificar parte de un recurso, dejando el resto como estaba. Y DELETE: borrar un recurso."

"Eso es todo el repertorio que vas a necesitar el noventa y cinco por ciento del tiempo. Existen otros — OPTIONS, HEAD, TRACE — pero los vas a tocar muy de vez en cuando o nunca. Cuando le dictás un endpoint a la IA, el primer paso es elegir uno de estos cinco. Si no podés elegir cuál corresponde, el contrato no arranca."

Status codes: 2xx / 4xx / 5xx

Lo primero que dice el servidor es un número de tres dígitos. El primer dígito te dice de qué tipo es la respuesta.

2xx

todo bien

  • 200 OK — GET que devolvió lo pedido
  • 201 creado — POST que creó algo
  • 204 sin contenido — DELETE
4xx

vos, cliente, te equivocaste

  • 400 mal pedido — body sin la forma esperada
  • 404 no existe — el path o el recurso no aparecen

401 y 403 (autenticación y permisos) → semana 7

5xx

yo, servidor, me caí

  • 500 algo se rompió adentro
  • 503 servicio no disponible

Las tres familias de status codes, una por click. El forward a semana 7 evita teach-ahead de auth.

"Cuando el servidor te responde, lo primero que dice es un número de tres dígitos. Está agrupado por familias, y cada familia te cuenta de qué tipo es la respuesta. Tres familias importan."

"Familia dos-equis-equis: todo bien. Doscientos OK, la respuesta clásica de un GET que devolvió lo que pediste. Doscientos uno creado, lo típico de un POST que creó un recurso nuevo. Doscientos cuatro sin contenido, lo típico de un DELETE — el recurso se borró y no hay nada más para devolver."

"Familia cuatro-equis-equis: vos, cliente, te equivocaste. El pedido llegó al servidor pero algo del lado del cliente está mal. Cuatrocientos mal pedido — mandaste un body que no tiene la forma que el servidor esperaba. Cuatrocientos cuatro no existe — el path o el recurso al que apuntaste no están. Acá también viven 401 y 403, autenticación y permisos, pero esos los vemos en semana 7. Hoy no los tocamos."

"Familia cinco-equis-equis: yo, servidor, me caí. El pedido estaba bien armado pero el servidor no pudo cumplirlo. Quinientos: algo se rompió adentro, típicamente una excepción no manejada en el código. Quinientos tres: servicio no disponible, el servidor está caído o saturado."

Todo es un recurso

Todo lo que la API expone es un sustantivo. Las URLs nombran sustantivos, no acciones.

/projects colección — todos los proyectos
/projects/4 item — el proyecto cuatro
/projects/4/tasks/17 recurso anidado — la tarea 17 de ese proyecto

El method dice qué hacer. La URL dice sobre qué. La acción viaja en el method, no en la URL.

La idea central de REST. Tres formas de URL — colección, item, recurso anidado — y el mantra que separa method de URL.

"La idea central de REST se puede decir en una frase: todo lo que la API expone es un recurso, un sustantivo. Un proyecto, una tarea, un usuario, un mensaje. Las URLs nombran sustantivos, no acciones. Eso es lo que descarta barra borrar-tarea y deja barra tasks barra diecisiete."

"Los recursos vienen en dos formas, más una variante. Una colección es el conjunto entero: barra projects significa 'todos los proyectos'. Un item es uno solo: barra projects barra cuatro es 'el proyecto cuatro'. Y si un recurso solamente tiene sentido dentro de otro, anidás: barra projects barra cuatro barra tasks barra diecisiete es 'la tarea diecisiete del proyecto cuatro'. Tres formas, mismo principio: la URL es un camino que nombra cosas."

"Y acá está la regla que conviene memorizar. El method dice qué hacer; la URL dice sobre qué. GET barra projects barra cuatro lee el proyecto cuatro. DELETE barra projects barra cuatro lo borra. PUT barra projects barra cuatro lo reemplaza. La URL no cambia según la acción. La acción viaja siempre en el method. Si te encontrás escribiendo una URL con un method adentro — borrar, crear, actualizar — algo está mal."

Regla 1 — Recursos en plural

Correcto

GET  /users
GET  /users/42
GET  /projects
GET  /projects/4

Inconsistente

GET  /user
GET  /users/42
GET  /project

Regla 2 — Jerarquía cuando hay relación

Anidado

/projects/4/tasks

La tarea pertenece al proyecto. Sin proyecto no existe.

Plano con filtro

/tasks?project=4

La tarea vive sola. El proyecto la modifica como filtro.

Anidás cuando la pertenencia es estructural, no cuando es opcional.

Regla 2: cómo decidir entre URL anidada y URL plana con query param. La distinción es semántica, no estética.

"Segunda regla. Cuando hay una relación entre recursos, usás jerarquía. Pero no siempre — depende de cómo es la relación. Las dos opciones de la pantalla son las dos formas legítimas, y se eligen por significado."

"Izquierda: barra projects barra cuatro barra tasks. Anidado. Esta forma dice 'las tareas del proyecto cuatro' como una afirmación estructural. La tarea pertenece al proyecto: si borrás el proyecto, las tareas no tienen sentido. La URL refleja esa dependencia."

"Derecha: barra tasks con un query param project igual a cuatro. Plano. Esta forma dice 'todas las tareas, filtradas por proyecto'. La tarea es un recurso de primera clase que existe sola; el proyecto es una propiedad que se usa para filtrar. Una tarea puede no pertenecer a ningún proyecto y seguir siendo tarea."

"La regla mental que te lleva la decisión: anidás cuando la pertenencia es estructural, no cuando es opcional. Si una tarea no flota suelta, anidás. Si una tarea puede vivir por sí misma, mantenés plano y filtrás. Es la primera decisión arquitectónica que vas a tener que dictarle a la IA cuando le pidas un endpoint nuevo."

Si era POST, no: retentar te crea un proyecto duplicado, una transferencia duplicada, una notificación duplicada. Tres razones para una base de datos

Persistencia

Eficiencia de consulta

Los errores tienen dueño

El primer dígito del código te dice de qué lado está el problema antes de mirar nada más.

4xx

Vos, cliente, te equivocaste.

  • 400 mal pedido — body con la forma equivocada
  • 404 no existe — el path o el recurso no aparecen
  • 422 datos inválidos — el body tiene la forma pero un campo no pasa la validación

401 y 403 (autenticación y permisos) → semana 7

5xx

Yo, servidor, me caí.

  • 500 algo se rompió adentro — excepción no manejada
  • 503 servicio no disponible — caído o saturado

Si arranca con 4, mirá tu pedido. Si arranca con 5, mirá los logs. Empezar por el lado equivocado es perder media hora.

Recuperar el mantra de §2.7 y aplicarlo. La línea de forward a semana 7 evita teach-ahead de auth.

"En la sección dos vimos las tres familias de códigos por arriba. Acá las usamos para diagnosticar. Los errores tienen dueño, y el primer dígito te dice cuál."

"Familia cuatro-equis-equis: vos, cliente, te equivocaste. El pedido llegó al servidor, lo entendió, y te dijo 'no, así no'. Cuatrocientos: el body no tiene la forma esperada. Cuatrocientos cuatro: el path o el recurso no existen. Cuatrocientos veintidós, que en la sección dos no nombramos: el body tiene la forma correcta pero un campo concreto no pasa la validación — por ejemplo mandaste un email que no es un email. Ahí también viven 401 y 403, autenticación y permisos, pero esos los vemos en semana 7."

"Familia cinco-equis-equis: yo, servidor, me caí. El pedido estaba bien armado pero el servidor no pudo cumplirlo. Quinientos: algo se rompió adentro, casi siempre una excepción no manejada en el código. Quinientos tres: el servicio está caído o saturado."

"Y el mantra que vamos a usar toda la sección: si arranca con cuatro, mirá tu pedido — la pelota está en tu cancha. Si arranca con cinco, mirá los logs del servidor — la pelota está del otro lado. "La fuente de verdad cuando algo se rompe son los logs. Un log es una línea que el servidor escribe cada vez que pasa algo: cada request que entra, cada error que ocurre, cada respuesta que sale. El stack trace: el camino al bug

Un stack trace es la cadena de llamadas que llevó al error, de la más externa a la más interna. No se lee de arriba a abajo. 

Traceback (most recent call last):
  File "uvicorn/server.py", line 245, in handle
  File "fastapi/routing.py", line 218, in app
  File "app/routes/tasks.py", line 14, in create_task
    db.execute("INSERT INTO tasks (title) VALUES (?)", (body.title,))
  File "sqlite3.py", line 312, in execute
sqlite3.IntegrityError: NOT NULL constraint failed: tasks.project_id

El síntoma grita al final. La causa vive en la primera línea de código que escribiste vos.

Tres estados: vista completa, después highlight de las líneas 4 y 5 (tu archivo + el INSERT), después highlight de la línea 7 (el síntoma).

Cómo leer un stack trace: ignorar el framework, encontrar la primera línea que menciona código tuyo, leer el síntoma al final. Aside sobre observabilidad ya plantado en slide anterior.

"Cuando el servidor revienta, además de la línea de error en los logs aparece un stack trace. Es la cadena de llamadas a funciones que llevó al error, listada desde la más externa hasta la más interna. Y la trampa para principiantes es leerlo arriba a abajo como si fuera prosa. No se lee así. La línea que importa suele ser la primera que menciona código tuyo, no del framework."

"Mirá las dos líneas iluminadas. 'app barra routes barra tasks.py línea catorce, función create_task'. Esa es la primera línea que apunta a tu carpeta app. Las dos de arriba — uvicorn, fastapi — son código de framework, ahí no vas a encontrar nada que arreglar; los frameworks rara vez son el problema. La que importa es la tuya. Y debajo tiene el detalle: el INSERT contra la tabla tasks que solamente le pasa title como columna, sin project_id. Ahí ves la causa concreta: el handler escribió un INSERT incompleto."

"Y abajo de todo, la última línea iluminada: sqlite3.IntegrityError, NOT NULL constraint failed, tasks.project_id. Ese es el síntoma — lo que la base de datos terminó gritando cuando se topó con el INSERT incompleto. Fijate el patrón, porque se repite siempre: el síntoma grita al final, la causa vive en la primera línea de código que escribiste vos. O que la IA escribió por vos. Un stack trace es un mapa con flecha: ignorá el framework, buscá tu archivo, leé qué línea de tu código provocó el grito final."

FastAPI es una elección, no la única

Node + Express

JavaScript en el servidor. Popular en startups donde el front-end ya manda; compartir lenguaje cliente↔servidor es atractivo para un equipo chico.

Django

Python opinionated. Trae todo: ORM, admin, migraciones, autenticación. Si seguís sus convenciones, escribís muy poco código.

Flask

Python minimal. Filosofía cercana a FastAPI pero sin schemas tipados de fábrica: lo armás pieza por pieza.

Go (Gin / Fiber)

Compilado, rápido, buen consumo de memoria. Popular para infra y servicios donde el costo por request importa.

Ruby on Rails

Convención sobre configuración llevada al extremo. Sigue vivo en muchas apps reales en producción.

El punto no es aprenderlas. Es saber que FastAPI es una elección: la elegimos por tipos modernos, schemas integrados, documentación automática y baja ceremonia.
