Anatomía de un par request / response

Cliente arma un request. Servidor arma una response. Entre los dos cabe casi todo lo que hace internet hoy. 

GET /projects/4 HTTP/1.1
Host: api.example.com

— — — respuesta — — —

HTTP/1.1 200 OK
Content-Type: application/json

{ "id": 4, "name": "Tesis" }

Request — cuatro partes

method · path · headers · body opcional

Response — tres partes

status code · headers · body

Anatomía concreta. Se lee el bloque línea por línea mientras está en pantalla, sin fragments. Es el zoom-in al paso 1 del diagrama de §1.4 — donde HTTP desempaca el pedido.

"Ese idioma común se llama HTTP, y es un protocolo de pedido y respuesta. El cliente arma un request, el servidor arma una response, y entre los dos cabe casi todo lo que hace internet hoy. En el diagrama de §1.4 vimos un POST con body — ahora un GET, más simple, para mirar las partes con calma."

"Arriba, el request, el pedido. Tiene cuatro partes. Primero el method: GET — uno de los cinco que importan, ya viene definido por el protocolo, no es una palabra que vos inventás. Después el path: barra projects barra cuatro. Es a qué cosa le estás hablando — querés el proyecto número cuatro. Después HTTP barra uno punto uno, que es la versión del protocolo. Y abajo los headers (metadatos). Acá tenemos uno solo: Host, que dice a qué dominio le estoy pidiendo — api.example.com. Este request no lleva body, porque los GET típicamente no llevan: solo piden información, no mandan contenido."

"Abajo, la response. Tres partes. Primero el status code: doscientos OK. Eso quiere decir 'todo bien, acá tenés lo que pediste'. Después los headers: Content-Type application/json, que dice 'lo que te mando es JSON'. Y abajo el body — el contenido en sí, en este caso un objeto con id cuatro y nombre Tesis."

"Ese ida y vuelta — request, response — repetido millones de veces por segundo, es internet. Pasa cada vez que el cliente (un navegador, una app, un curl) le habla al servidor. Y el vocabulario para describirlo es exactamente lo que tenés en pantalla. Cuando le dictés un endpoint a la IA, vas a tener que nombrar estas piezas — el method, el path, qué headers esperás, qué viene en el body. Si no las nombrás, el modelo elige por vos."

La palabra method engaña: no son acciones libres que componés vos. Son un repertorio cerrado — operaciones que se repiten en toda API.

Ese es todo el repertorio que vas a necesitar el 95% del tiempo.

Cinco methods como repertorio cerrado. La aclaración terminológica importa: no son acciones libres. En §1.4 ya vimos que el method dice qué hacer con la cosa que nombra el path — acá listamos los cinco que importan.

"HTTP define un puñado de methods. La palabra engaña un poco: no son acciones que vos componés libremente. Son un repertorio cerrado — las pocas operaciones que aparecen en toda API. Como dijimos en el diagrama de §1.4, el method dice qué hacer con la cosa que nombra el path."

"Cinco que importan. GET: leer un recurso, sin efectos. Pedís información y nada cambia del lado del servidor. POST: crear un recurso nuevo (como el POST barra projects barra cuatro barra tasks del diagrama). Mandás contenido y aparece algo que antes no existía. PUT: reemplazar un recurso entero por la versión que mandás — la versión vieja desaparece. PATCH: modificar parte de un recurso, dejando el resto como estaba. Y DELETE: borrar un recurso."

"Eso es todo el repertorio que vas a necesitar el noventa y cinco por ciento del tiempo. Existen otros — OPTIONS, HEAD, TRACE — pero los vas a tocar muy de vez en cuando o nunca. Cuando le dictás un endpoint a la IA, el primer paso es elegir uno de estos cinco. Si no podés elegir cuál corresponde, el contract no arranca."

Status codes: 2xx / 4xx / 5xx

Lo primero que dice el servidor es un número de tres dígitos. El primer dígito te dice de qué tipo es la respuesta.

2xx

todo bien

  • 200 OK — GET que devolvió lo pedido
  • 201 creado — POST que creó algo
  • 204 sin contenido — DELETE
4xx

vos, cliente, te equivocaste

  • 400 mal pedido — body sin la forma esperada
  • 404 no existe — el path o el recurso no aparecen

401 y 403 (autenticación y permisos) → semana 7

5xx

yo, servidor, me caí

  • 500 algo se rompió adentro
  • 503 servicio no disponible

Tres clicks: revelar 2xx, después 4xx, después 5xx.

Las tres familias de status codes, una por click. El forward a semana 7 evita teach-ahead de auth. Las ramas Response (2xx) y Errores (5xx) del diagrama de §1.4 son justamente dos de estas familias — acá las nombramos en detalle y agregamos la del medio (4xx).

"Cuando el servidor te responde, lo primero que dice es un número de tres dígitos. Está agrupado por familias, y cada familia te cuenta de qué tipo es la respuesta. En el diagrama de §1.4 ya vieron dos ramas: Response 2xx (verde) y Errores 5xx (rojo). Ahora las desarmamos en detalle, y aparece una tercera familia que no estaba en el diagrama: 4xx."

"Familia dos-equis-equis: todo bien. Doscientos OK, la respuesta clásica de un GET que devolvió lo que pediste. Doscientos uno creado, lo típico de un POST que creó un recurso nuevo (el caso del diagrama, cuando el dado salía verde). Doscientos cuatro sin contenido, lo típico de un DELETE — el recurso se borró y no hay nada más para devolver."

"Familia cuatro-equis-equis: vos, cliente, te equivocaste. El pedido llegó al servidor pero algo del lado del cliente está mal. Cuatrocientos mal pedido — mandaste un body que no tiene la forma que el servidor esperaba. Cuatrocientos cuatro no existe — el path o el recurso al que apuntaste no están. Acá también viven 401 y 403, autenticación y permisos, pero esos los vemos en semana 7. Hoy no los tocamos."

"Familia cinco-equis-equis: yo, servidor, me caí. El pedido estaba bien armado pero el servidor no pudo cumplirlo (la rama roja del diagrama, cuando la base rechazaba el INSERT). Quinientos: algo se rompió adentro, típicamente una excepción no manejada en el código. Quinientos tres: servicio no disponible, el servidor está caído o saturado."

HTTP

El transporte. Define cómo viaja un pedido y cómo viene la respuesta.

Todo es un recurso

REST modela cosas, no acciones. Todo lo que la API expone es un recurso concreto.

/projects colección — todos los proyectos
/projects/4 item — el proyecto cuatro
/projects/4/tasks/17 recurso anidado — la tarea 17 de ese proyecto

El method dice qué hacer. La URL dice sobre qué. La acción viaja en el method, no en la URL.

La idea central de REST. Tres formas de URL — colección, item, recurso anidado — y el mantra que separa method de URL.

"La idea central de REST cabe en una sola frase, y vale la pena aprenderla así: REST modela cosas, no acciones. Todo lo que la API expone es un recurso — algo concreto. Y si un recurso solamente tiene sentido dentro de otro, anidás: barra projects barra cuatro barra tasks barra diecisiete es 'la tarea diecisiete del proyecto cuatro'. "Y acá está la regla que conviene memorizar. El method dice qué hacer; la URL dice sobre qué. GET barra projects barra cuatro lee el proyecto cuatro. DELETE barra projects barra cuatro lo borra. Si te encontrás escribiendo una URL con un verbo adentro (borrar, crear, actualizar), algo está mal."

Regla 2 — Jerarquía cuando hay relación

Anidado

/projects/4/tasks

La tarea pertenece al proyecto. Sin proyecto no existe.

Plano con filtro

/tasks?project=4

La tarea vive sola. El proyecto la modifica como filtro.

Anidás cuando la pertenencia es estructural, no cuando es opcional.

Regla 2: cómo decidir entre URL anidada y URL plana con query param. La distinción es semántica, no estética.

"Segunda regla. Cuando hay una relación entre recursos, usás jerarquía. Pero no siempre — depende de cómo es la relación. Las dos opciones de la pantalla son las dos formas legítimas, y se eligen por significado."

"Izquierda: barra projects barra cuatro barra tasks. Es exactamente el path que vimos viajar en el ejemplo del recorrido. Anidado. Esta forma dice 'las tareas del proyecto cuatro' como una afirmación estructural. La tarea pertenece al proyecto: si borrás el proyecto, las tareas no tienen sentido. La URL refleja esa dependencia."

"Derecha: barra tasks con un query param project igual a cuatro. Plano. Esta forma dice 'todas las tareas, filtradas por proyecto'. La tarea es un recurso de primera clase que existe sola; el proyecto es una propiedad que se usa para filtrar. Una tarea puede no pertenecer a ningún proyecto y seguir siendo tarea."

"La regla mental que te lleva la decisión: anidás cuando la pertenencia es estructural, no cuando es opcional. Si una tarea no flota suelta, anidás. Si una tarea puede vivir por sí misma, mantenés plano y filtrás. Es la primera decisión arquitectónica que vas a tener que dictarle a la IA cuando le pidas un endpoint nuevo."

PATCH depende: si tu PATCH suma uno a un contador, dos llamadas suman dos; si tu PATCH fija un campo a un valor, dos llamadas dan el mismo resultado. "Click. Acá está la consecuencia. Imaginá que mandás un request, la red se cae y no sabés si el servidor lo recibió. ¿Lo mandás de nuevo? Si era POST, no: reintentar te crea un proyecto duplicado, una transferencia duplicada, una notificación duplicada. La idempotencia es lo que te dice si tu cliente puede tener un botón 'reintentar' o si necesita lógica más cuidadosa. Cuando le pidas a la IA un cliente que tolere fallos de red, el primer dato que tiene que conocer es qué de tu API es idempotente."

Tres razones para una base de datos

Persistencia

Sobrevive a reinicios, despliegues y sesiones que se cierran. Lo escrito sigue ahí mañana porque vive en disco, no en memoria.

Eficiencia de consulta

Integridad

Los errores tienen dueño

El primer dígito del código te dice de qué lado está el problema antes de mirar nada más.

4xx

Vos, cliente, te equivocaste.

  • 400 mal pedido — body con la forma equivocada
  • 404 no existe — el path o el recurso no aparecen
  • 422 datos inválidos — el body tiene la forma pero un campo no pasa la validación

401 y 403 (autenticación y permisos) → semana 7

5xx

Yo, servidor, me caí.

  • 500 algo se rompió adentro — excepción no manejada
  • 503 servicio no disponible — caído o saturado

Si arranca con 4, mirá tu pedido. Si arranca con 5, mirá los logs. Empezar por el lado equivocado es perder media hora.

  File "sqlite3.py", line 312, in execute

"Cuando el servidor revienta, además de la línea de error en los logs aparece un stack trace. El bloque de contrato, en dos formatos

Contrato — §4

Texto informal

POST /projects/{id}/tasks
  entrada (body):
    title       string, requerido
    due_date    fecha ISO, opcional
  salida 201 (creado):
    { id, title, due_date, project_id }
  salida 400 (datos inválidos):
    { error: "title es requerido" }
  salida 404 (proyecto no existe):
    { error: "no se encontró el proyecto {id}" }

OpenAPI 3.1

YAML, machine-readable

paths:
  /projects/{id}/tasks:
    post:
      requestBody:
        required: true
        content:
          application/json:
            schema:
              type: object
              required: [title]
              properties:
                title:    { type: string }
                due_date: { type: string, format: date, nullable: true }
      responses:
        '201':
          description: Created
          content: { application/json: { schema: { $ref: '#/components/schemas/Task' } } }
        '400':
          description: title es requerido
        '404':
          description: proyecto no existe

Misma información, dos formatos. La derecha es la que la IA prefiere — y la que el equipo versiona mañana.

Click 4: ilumina salida 201 ↔ responses '201'. Click 5: ilumina las salidas 400/404 ↔ '400' y '404' (errores).

"Schema-out — lo que llamábamos 'salida 201' / 'salida 400' / 'salida 404' en OpenAPI son entries dentro de responses, con la key igual al status code. "Cinco piezas, cinco mappings, una correspondencia exacta. La derecha no agrega información nueva; cierra ambigüedades. Y como es un archivo machine-readable, mucho de lo que vimos manualmente lo va a poder hacer una herramienta."

Por qué importa que sea un archivo

Razón 1

Menos ambigüedad

Tipos, formatos, requeridos, nullables: fijados. Lo que en un prompt es opinable, en el yaml está escrito. La IA no improvisa.

Razón 2

Codegen real

Del mismo archivo: docs interactivas (Swagger UI), clientes en cualquier lenguaje, mocks, tests de contrato. Una sola fuente, muchos artefactos.

Razón 3

Durabilidad

No es un prompt que se pierde. Es un archivo en el repo: se versiona, se compara entre branches, se vuelve a dictar.

El supervisor arquitectónico ahora dirige un artefacto durable.

Cierre de §7. Tres razones por las que el contrato hecho archivo desbloquea cosas que un prompt no puede.

"Tres razones por las que importa que el contrato sea un archivo y no un prompt. Una por una."

"Primera, ambigüedad. El yaml fija tipos, formatos como format: date, requeridos, nullables. Cada uno de esos detalles, en un prompt vago, lo decide la IA. En el yaml lo decidís vos. Es la misma idea de 'vago vs específico' que cerró las cinco secciones anteriores, ahora hecha archivo."

"Segunda, codegen. Y esto es lo que vamos a ver renderizado en el demo. Del mismo openapi.yaml salen docs interactivas — eso es Swagger UI, lo van a ver en pantalla en cinco minutos. Salen clientes generados en cualquier lenguaje (Python, TypeScript, Go), salen mocks que el frontend puede consumir mientras el backend no existe, salen tests que verifican que la implementación cumple el contrato. Una sola fuente, muchos artefactos derivados. Eso es codegen real, no abstracción."

"Tercera, durabilidad. Un prompt vive en una conversación. Cuando cerrás la pestaña se pierde, o se diluye en mil mensajes después.
